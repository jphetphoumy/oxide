/switch — Agent Switching Feature

Implementation plan for in-session agent switching via slash command

oxide v0.4.5 2026-06-03 ratatui + Dust API

Overview

Currently the Oxide TUI locks to a single agent for the entire session (resolved at startup from OXIDE_AGENT_ID / config / default "dust"). This plan adds a /switch slash command that lets users list available agents and switch mid-conversation without restarting.

User experience

  • Type /switch → see a filterable list of agents
  • Arrow-key or type to filter, Enter to select
  • Status bar updates to reflect new agent
  • Next message is sent to the new agent (same conversation)

Non-goals

  • Creating / editing agent configurations
  • Starting a new conversation on switch
  • Persisting agent choice to config file
  • Multi-agent mentions in a single message

Dust API Surface

Two endpoints are relevant:

GET /api/v1/w/{wId}/assistant/agent_configurations

Lists all accessible agents. We use view=list (authenticated default) to get the user's active agents.

Response shape:

{
  "agentConfigurations": [
    {
      "sId": "abc123",
      "name": "dust",
      "description": "General-purpose assistant",
      "pictureUrl": "https://...",
      "status": "active",
      "scope": "workspace"
    },
    ...
  ]
}
GET /api/v1/w/{wId}/assistant/agent_configurations/search?q={query}

Searches agents by name. Useful if the workspace has many agents — we can do client-side filtering on the cached list instead, and fall back to this endpoint only if needed.

Key fields we need from AgentConfiguration:

FieldTypeUsage
sIdstringAgent identifier — passed as configurationId in mentions and stored in DustClient.agent_id
namestringDisplay name shown in the picker and status bar
descriptionstringShown as secondary text in the picker
pictureUrlstringNot used (TUI has no images)
scopestringGroup agents: workspace vs published vs global

Architecture

App mode state machine

The app gains a mode enum. The TUI main loop dispatches events differently based on the current mode:

enum AppMode {
    Chat,                     // current behavior
    AgentPicker(PickerState), // /switch overlay
}

struct PickerState {
    agents: Vec<AgentInfo>,   // fetched from API
    filter: String,            // live search text
    selected: usize,           // cursor index in filtered list
    loading: bool,             // true while API call in-flight
}

struct AgentInfo {
    s_id: String,
    name: String,
    description: String,
    scope: String,
}

User flow

1 User types /switch + Enter
↓
2 Handler detects slash command, switches to AgentPicker mode
↓
3 Async task calls GET /agent_configurations?view=list
↓
4 Overlay renders: search box + scrollable agent list
↓
5 User filters / arrows / selects with Enter (or Esc to cancel)
↓
6 App updates agent_id + agent_name, returns to Chat mode
↓
7 Status bar reflects new agent; next message mentions new agent

Implementation Steps

Step 1 — API types + client method

Add the Dust API types and list_agents() to the client. This is pure backend work, fully testable in isolation.

  • Add AgentInfo and ListAgentsResponse to dust/types.rs
  • Add DustClient::list_agents() in dust/client.rs
  • Add DustClient::set_agent(&mut self, id, name)
  • Unit tests for deserialization

Step 2 — App mode + state

Introduce AppMode, PickerState, and agent switching on the App struct.

  • Add AppMode enum, PickerState struct to app.rs
  • Add mode field + accessors: mode(), enter_picker(), exit_picker()
  • Add switch_agent(id, name) — updates agent_name, pushes system message "Switched to {name}"
  • Add picker state mutation: set_agents(), update_filter(), move_selection()
  • Unit tests for all state transitions

Step 3 — Slash command detection + picker keybindings

Wire up the handler to intercept /switch and handle picker input.

  • In apply_action(), on Submit: if content is /switch, enter picker mode instead of sending to Dust
  • New PickerAction enum: MoveUp, MoveDown, Select, Cancel, Type(char), Backspace
  • New handle_picker_key() function mapping keys to PickerAction
  • New apply_picker_action() function applying actions to PickerState
  • Return a new ActionOutcome variant for agent selection
  • Unit tests for slash command detection + picker key mapping

Step 4 — Picker UI widget

Build the overlay widget in a new ui/picker.rs module.

  • Centered popup: Clear + Block with border + title "Switch Agent"
  • Search input line at top with cursor
  • Scrollable List widget below with filtered agents
  • Each item: name (white/bold) — description (dark gray, truncated)
  • Highlight style: accent background on selected row
  • Loading state: "Fetching agents..." centered
  • Empty state: "No agents match" centered
  • Footer hint: "↑↓ navigate Enter select Esc cancel"

Step 5 — Main loop integration

Wire everything together in main.rs.

  • New mpsc::unbounded_channel<Vec<AgentInfo>> for async agent list results
  • On /switch submit: spawn client.list_agents() task, send result through channel
  • In tokio::select!: add branch for agent list channel
  • On agent list received: update PickerState with agents, set loading = false
  • On agent selected: call app.switch_agent(), update DustClient agent, return to Chat
  • Draw: conditionally render picker overlay based on app.mode()

Step 6 — Integration testing

TUI integration tests using tmux.

  • Test: type /switch + Enter → picker overlay appears
  • Test: Esc closes picker, returns to chat
  • Test: status bar shows updated agent name after switch

Key Design Decisions

Same conversation, different agent

Switching agents does not start a new conversation. The Dust API supports mentioning different agents within the same conversation — the mentions[].configurationId field in each message body controls which agent responds. This means:

  • Context from previous messages is preserved
  • The user can switch freely without losing history
  • Each message can target a different agent

Client-side filtering

We fetch the full agent list once when the picker opens and filter client-side. The search API endpoint (?q=) exists as a fallback but most workspaces have <100 agents, making a single cached fetch more responsive than per-keystroke API calls.

DustClient mutability

Currently DustClient is cloned per message-send task. To support switching, we wrap agent_id in the client with a setter, or — simpler — we pass the agent_id directly to send_message_flow() instead of reading it from self. This avoids shared mutable state entirely. The client already accepts agent_id as a parameter in create_conversation() and post_message().

Future Extensions

  • /switch <name> — direct switch without picker if agent name is unambiguous
  • Cache agent list for the session, refresh on demand
  • Show current agent as highlighted in the picker
  • Agent favorites / recently used sorting
  • Generic slash command framework for /help, /clear, /new, etc.
