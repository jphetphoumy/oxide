Slash Commands & Agent Switching

Inline command menu with / trigger, Tab completion, and /switch agent picker overlay

oxide v0.4.5 2026-06-03 ratatui + Dust API ADR-0008

Overview

This plan covers two interconnected features:

  1. Slash command framework — typing / in the input shows an inline autocomplete menu listing available commands. Typing further filters the list. Tab completes the top match. Enter submits the completed command.
  2. /switch command — submitting /switch opens a centered overlay to pick a different Dust agent, preserving the current conversation.

User experience

  • Type / → inline menu appears below input with available commands
  • Continue typing to filter — /sw narrows to /switch
  • Tab completes the top match into the input
  • Enter submits the completed command
  • /switch → centered overlay with agent list, filter, arrow keys, Enter to select
  • Esc dismisses either the menu or the picker

Non-goals

  • Creating / editing agent configurations
  • Starting a new conversation on agent switch
  • Persisting agent choice to config file
  • Slash commands with arguments (e.g. /switch <name>)

Dust API Surface

Two endpoints are relevant (used by /switch):

GET /api/v1/w/{wId}/assistant/agent_configurations?view=list

Lists all accessible agents for the authenticated user.

{
  "agentConfigurations": [
    { "sId": "abc123", "name": "dust", "description": "General-purpose assistant", "scope": "workspace" },
    ...
  ]
}
GET /api/v1/w/{wId}/assistant/agent_configurations/search?q={query}

Server-side name search. Available as fallback; we prefer client-side filtering.

Architecture

Slash command registry

Commands are defined in a static registry. Each entry has a name, description, and the action to perform when submitted. This makes it trivial to add new commands later.

struct SlashCommandDef {
    name: &'static str,        // e.g. "switch"
    slash_name: &'static str,  // e.g. "/switch" (for Tab completion)
    description: &'static str, // e.g. "Switch to a different agent"
}

const COMMANDS: &[SlashCommandDef] = &[
    SlashCommandDef { name: "switch", slash_name: "/switch",
                      description: "Switch to a different agent" },
    // future: { name: "clear", ... },
];

App mode state machine

enum AppMode {
    Chat,                     // normal input — if text starts with "/" show command menu
    Picker(PickerState),      // /switch overlay (already implemented)
}

// The command menu is NOT a mode — it's a UI hint rendered when
// the input buffer starts with "/" while in Chat mode.

Input-driven command menu

The command menu is not a separate app mode. It is rendered as an inline popup whenever the input buffer starts with / during Chat mode. This keeps the state machine simple:

  • Rendering checks input.content().starts_with('/')
  • The filter text is everything after /
  • Tab inserts the top match's full name (replacing current input)
  • Enter on a completed command triggers the command action
  • Esc or Backspace past / dismisses the menu naturally

User flow — command menu

1 User types /
↓
2 Inline menu appears above input with matching commands
↓
3 User types more chars to filter (e.g. /sw)
↓
4 Tab → top match auto-completed into input (/switch)
↓
5 Enter → command submitted
↓
6 Handler dispatches to command action (e.g. open agent picker)

User flow — agent picker (unchanged)

1 /switch submitted → enter Picker mode + fetch agents
↓
2 Centered overlay with search + scrollable agent list
↓
3 Filter / arrow / Enter to select (Esc to cancel)
↓
4 App updates agent_id + agent_name, returns to Chat

Key Design Decisions

Menu is not a mode

The command menu is a render-time decision based on the input buffer content, not a separate AppMode. This keeps the state machine simple (only Chat and Picker) and avoids needing to track menu open/close state. The menu appears naturally when input starts with / and disappears when it doesn't. See ADR-0008.

Same conversation on agent switch

Switching agents does not start a new conversation. The Dust API supports mentioning different agents within the same conversation via the mentions[].configurationId field. Context is preserved; each message can target a different agent.

Client-side filtering

Both the command menu and the agent picker use client-side filtering. The command list is static. The agent list is fetched once per picker open. This is more responsive than per-keystroke API calls.

Future Extensions

  • /switch <name> — direct switch without picker if agent name is unambiguous
  • /clear — clear conversation history
  • /new — start a new conversation
  • /help — show keybindings and commands
  • Cache agent list for the session, refresh on demand
  • Down/Up arrow navigation within the command menu
  • Show current agent as highlighted in the picker
