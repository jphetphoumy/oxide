MCP Support & Bash Tool

Model Context Protocol client, built-in bash tool, tool-call UX, and Dust streaming integration

issue #11 2026-06-13 oxide · ratatui · MCP JSON-RPC

Overview

This plan adds Model Context Protocol (MCP) support to Oxide. MCP lets AI agents invoke locally-defined tools — starting with a bash shell tool — and receive the results back before continuing to generate their response. Oxide acts as an MCP client: it discovers tools from configured MCP servers (or its own built-in servers), exposes them to the Dust agent when sending a message, and handles the round-trip execution loop.

What this adds

  • Config block for MCP servers in config.toml
  • MCP client: stdio JSON-RPC process management + tool discovery
  • Built-in bash MCP server (no external process needed)
  • Tool-call rendering in the chat transcript
  • Safety confirmation modal (approve / deny per call)
  • Dust streaming: parse tool_use events + submit results

Non-goals

  • GUI MCP server editor
  • Remote MCP servers (HTTP transport) — stdio only for now
  • Tool result streaming (results are sent atomically)
  • Per-tool permission policies beyond approve/deny
  • Sandbox / container isolation for bash commands
  • Exposing Dust's built-in tools (search, data sources)

Configuration Format

MCP servers are declared in ~/.config/oxide/config.toml. The built-in bash server needs no command — it runs in-process.

[mcp]
auto_approve = false   # if true, skip confirmation modal for all tools

# built-in bash tool — no command field needed
[[mcp.servers]]
name    = "bash"
builtin = "bash"

# external MCP server
[[mcp.servers]]
name    = "my-tools"
command = "npx"
args    = ["-y", "@my-org/mcp-server"]
env     = { MY_API_KEY = "secret" }

The auto_approve flag defaults to false. When true all tool calls execute without confirmation — useful for trusted, non-destructive toolsets.

MCP Protocol Overview

MCP uses JSON-RPC 2.0 over stdio. The client writes a request to the server's stdin; the server writes a response to its stdout. Messages are newline-delimited JSON.

Relevant JSON-RPC methods

DirectionMethodPurpose
client → serverinitializeHandshake; negotiate protocol version
client → servertools/listDiscover available tools and their JSON schemas
client → servertools/callExecute a tool with given arguments
server → clientresponseTool result (content array)

Tool discovery response shape

{
  "tools": [
    {
      "name": "bash",
      "description": "Run a bash command and return its stdout/stderr",
      "inputSchema": {
        "type": "object",
        "properties": {
          "command": { "type": "string", "description": "Shell command to execute" }
        },
        "required": ["command"]
      }
    }
  ]
}

Tool call result shape

{
  "content": [
    { "type": "text", "text": "total 48\ndrwxr-xr-x ..." }
  ],
  "isError": false
}

Dust API Integration

When creating a Dust message, available tools are listed in the request body. The SSE stream may then emit a tool_use event asking Oxide to run a tool. After execution, Oxide sends the tool result via a follow-up API call before the agent continues generating.

Message request with tools

POST /api/v1/w/{wId}/assistant/conversations/{cId}/messages
{
  "content":  [{ "type": "text", "value": "what's in my project?" }],
  "mentions": [{ "configurationId": "dust" }],
  "context":  { ... },
  "tools": [
    {
      "name":        "bash",
      "description": "Run a bash command",
      "inputSchema": { ... }
    }
  ]
}

SSE stream — tool_use event

event: agent_action_success
data: {
  "action": {
    "type":  "tool_use",
    "id":    "toolu_abc123",
    "name":  "bash",
    "input": { "command": "ls -la /home/user/project" }
  }
}

Tool result submission

POST /api/v1/w/{wId}/assistant/conversations/{cId}/tool_results
{
  "tool_use_id": "toolu_abc123",
  "content": [
    { "type": "text", "text": "total 48\ndrwxr-xr-x ..." }
  ]
}

After the result is submitted, the SSE stream resumes and the agent continues generating.

Data Flow — Tool Execution Round-Trip

1 App starts → McpManager::init() discovers tools from all servers
↓
2 User submits a message
↓
3 Dust API: POST message with tools: [...] array → start SSE stream
↓
4 SSE stream emits agent_generation_token (normal text) → rendered in transcript
↓
5 SSE stream emits tool_use event → parsed into ToolCall
↓
6 App enters ToolApproval mode → tool-call block shown with [y/n] prompt
↓
7a User presses y → Action::ApproveTool
↓
8a McpManager::call_tool() → bash executes command, returns ToolResult
↓
9a Dust API: submit_tool_result() → stream resumes
↓
10 App returns to Chat mode, agent continues generating

Step 7b: if the user presses n (deny), ToolResult { is_error: true, content: "denied by user" } is submitted and the agent is informed it cannot proceed with that tool call.

Key Design Decisions

ToolApproval is a full AppMode, not a render hint

Unlike the slash-command menu (which is render-time only), tool approval needs to block all other input — the streaming loop is paused waiting for a user decision. A full AppMode variant makes this explicit and prevents any accidental state transitions while waiting.

Built-in bash over embedded subprocess

The bash tool runs directly in Oxide via tokio::process::Command rather than as a separate MCP stdio server. This avoids a self-pipe round-trip and keeps the binary self-contained. External MCP servers still use the full stdio JSON-RPC client for third-party toolsets.

Streaming pauses on tool_use

The SSE stream is held open but not consumed while Oxide waits for approval and executes the tool. Dust's API design supports this — the agent waits for the tool result before generating more tokens. No buffering of future events is needed on the client side.

Security model: explicit approval only

The default is deny unless approved. auto_approve is opt-in and clearly documented as dangerous. There is no sandboxing of bash commands in this phase — that is a separate effort. The tool-call block always shows the exact command so the user can make an informed decision. Denial sends a structured error result back to the agent so it can respond gracefully rather than hanging.

Future Extensions

  • HTTP/SSE MCP transport for remote tool servers
  • Per-server auto_approve (e.g. always approve read-only tools)
  • Sandbox isolation for bash (nsjail, bubblewrap, Docker)
  • Edit mode — let user modify the command before approving
  • /tools slash command to list available MCP tools
  • Tool result truncation with scroll for large outputs
  • Persist tool session (carry tool list across conversation turns without re-discovery)
