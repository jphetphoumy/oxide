Oxide Subagent MCP Tool

Goal: Expose an oxide_agent MCP tool that spawns a fresh Dust conversation, sends a prompt to a chosen agent, streams the full response, and returns the final text as the tool result — enabling recursive agent orchestration from within a chat session.
Scope: New built-in tool registered in McpManager, new SubagentRunner module that wraps the existing DustClient message flow, TUI status indicator for in-flight subagent calls.
Out of scope: Multi-turn subagent conversations, streaming tokens back to the parent conversation in real time, per-tool configurable auto-approve (future extension).

1 Feature Overview

The feature mirrors Claude Code's Agent tool. When the Dust agent (or any MCP client connected to oxide's built-in MCP server) calls oxide_agent, oxide will:

  1. Create a brand-new Dust conversation with the target agent.
  2. Send the supplied prompt as the first (and only) user message.
  3. Wait for the agent to finish (streaming internally, discarding individual tokens).
  4. Return the final text as the MCP tool result string.

The call is fully async and non-blocking for the TUI event loop: it runs in a tokio::spawn task just like all other tool executions today.

Why a built-in tool, not an external MCP server?

The oxide_bash and oxide_skill tools are both built into McpManager and always available without extra configuration. The subagent tool needs authenticated DustClient access — which only the oxide process has — so it must also be built-in. It follows the exact same registration pattern.

2 Architecture

oxide_agent tool call arrives (from Dust SSE or MCP transport) | v McpManager::call_tool("oxide_agent", { prompt, agent_id?, description? }) | v SubagentRunner::run(client, prompt, agent_id) runs in tokio task | ├── POST /api/v1/w/{ws}/assistant/conversations (create conversation) ├── poll GET /api/v1/w/{ws}/assistant/conversations/{id} (wait for agent msg id) ├── open GET /api/sse/v1/w/{ws}/assistant/conversations/{id}/messages/{id}/events └── collect tokens until AgentMessageSuccess | v ToolResult { content: "...final agent response...", is_error: false } | v returned to MCP caller as JSON-RPC response

Integration point: two call paths

The oxide_agent tool can be invoked from two different paths, both already handled by the existing infrastructure:

  • Old inline path (DustEvent::ToolUse): Agent SSE emits a tool call, oxide executes it via McpManager::call_tool, submits result via /tool_results.
  • MCP transport path (DustEvent::McpToolUse): Dust calls oxide's registered MCP server via /mcp/requests SSE, oxide executes and posts via /mcp/results.

Both paths go through McpManager::call_tool, so registering oxide_agent once there covers both automatically.

DustClient availability in McpManager

Currently McpManager::call_tool is a pure function with no Dust client dependency — it handles bash commands and skill file reads. The subagent tool requires an authenticated DustClient. This is the key architectural decision: we must inject a DustClient (or its config) into McpManager at construction time.

Decision: Add an Option<DustClient> field to McpManager. The TUI path sets it from DustClient::from_env(). The mcp-server CLI path also has access to the same env/keyring, so it can construct one too.

3 New Module: src/mcp/subagent.rs

NEW FILE

This module contains SubagentRunner — the async function that orchestrates a complete one-shot Dust conversation and returns the final text.

// src/mcp/subagent.rs

use anyhow::{Result, anyhow};
use tokio::sync::mpsc;

use crate::dust::client::{DustClient, DustEvent};

/// Maximum nesting depth for local `oxide_agent` calls.
pub const MAX_SUBAGENT_DEPTH: u32 = 3;

/// Run a one-shot conversation with a Dust agent and collect the full response.
///
/// - Creates a fresh conversation (no reuse of the caller's conversation).
/// - Buffers tokens; does not forward streaming events to the TUI.
/// - Returns the complete agent response text on success.
/// - `depth` is the current nesting level (0 = top-level call from user session).
///   If `depth` >= `MAX_SUBAGENT_DEPTH`, returns an error immediately.
pub async fn run_subagent(
    client: &DustClient,
    prompt: String,
    agent_id: Option<String>,
    depth: u32,
) -> Result<String> {
    if depth >= MAX_SUBAGENT_DEPTH {
        return Err(anyhow!(
            "oxide_agent max depth ({MAX_SUBAGENT_DEPTH}) exceeded — cannot spawn further subagents"
        ));
    }

    // Clone the client and optionally override the agent ID.
    let mut subagent_client = client.clone();
    if let Some(id) = agent_id {
        subagent_client.set_agent(id);
    }

    // Use an unbounded channel — the subagent runner is the only sender.
    let (tx, mut rx) = mpsc::unbounded_channel::<DustEvent>();

    // Run the full message flow in the current task (already inside a tokio::spawn).
    // Pass empty active_skills — subagent conversations never inherit parent skills.
    // Pass depth + 1 so any nested oxide_agent calls know their depth.
    let flow = subagent_client
        .send_message_flow_with_skills(None, prompt, tx, &[], depth + 1)
        .await;

    // Drain all events from the channel (send_message_flow is done by now).
    let mut response_text = String::new();
    while let Ok(event) = rx.try_recv() {
        match event {
            DustEvent::Complete(Some(content), _) => {
                response_text = content;
            }
            DustEvent::Error(e) => {
                return Err(anyhow!("subagent error: {e}"));
            }
            DustEvent::Token(token, _) => {
                // Fallback: accumulate tokens if Complete doesn't include full content.
                response_text.push_str(&token);
            }
            _ => {}
        }
    }

    // Propagate any transport-level error from the flow itself.
    flow?;

    if response_text.is_empty() {
        return Err(anyhow!("subagent returned empty response"));
    }

    Ok(response_text)
}

Why drain after await, not concurrently?

send_message_flow_with_skills sends events and awaits to completion before returning. All events are in the channel by the time the future resolves. Draining with try_recv after the await is both simpler and sufficient — no need for a concurrent receiver task.

Depth threading

When run_subagent internally creates a Dust conversation and that conversation calls oxide_agent again, the nested call arrives via the normal MCP/SSE tool dispatch path. To propagate depth, run_subagent must construct a new McpManager instance (or a lightweight SubagentContext) with current_depth = depth + 1 and use that manager for the nested call. This ensures depth is accurately tracked regardless of how many levels deep the recursion goes.

Remote subagents (calls to external MCP servers that the Dust agent has configured in its own workspace) are not subject to the local depth counter. The depth limit only applies to calls that flow back through oxide's own oxide_agent built-in tool.

4.3 Handle oxide_agent in call_tool

    pub async fn call_tool(
        &mut self,
        tool_name: &str,
        input: serde_json::Value,
    ) -> Result<ToolResult> {
        // ... existing oxide_skill and oxide_bash branches ...

        if tool_name == "oxide_agent" {
            let prompt = input
                .get("prompt")
                .and_then(|v| v.as_str())
                .context("oxide_agent requires 'prompt'")?
                .to_string();

            let agent_id = input
                .get("agent_id")
                .and_then(|v| v.as_str())
                .map(ToString::to_string);

            // depth is threaded through from the call context; defaults to 0 if absent
            // (meaning this call came from the top-level session, not from a subagent).
            let depth = self.current_depth;

            let client = self
                .dust_client
                .as_ref()
                .context("oxide_agent: no Dust client available")?
                .clone();

            let result = crate::mcp::subagent::run_subagent(&client, prompt, agent_id, depth).await;

            return match result {
                Ok(text) => Ok(ToolResult {
                    content: text,
                    is_error: false,
                    tool_use_id: String::new(),
                }),
                Err(e) => Ok(ToolResult {
                    content: format!("subagent failed: {e}"),
                    is_error: true,
                    tool_use_id: String::new(),
                }),
            };
        }

        // ... rest of existing logic (external MCP clients) ...
    }

7 Dust API: What the Subagent Uses

The subagent runner reuses the existing send_message_flow_with_skills method which already orchestrates the full create-poll-stream flow. No new Dust API endpoints are needed.

MethodEndpointPurposeExisting?
POST /api/v1/w/{ws}/assistant/conversations Create fresh subagent conversation Yes — create_conversation_with_skills
GET /api/v1/w/{ws}/assistant/conversations/{id} Poll for agent message ID Yes — get_conversation
GET /api/sse/v1/w/{ws}/assistant/conversations/{id}/messages/{id}/events Stream agent response Yes — stream_events

Agent ID resolution for subagents

The agent_id parameter in the tool schema is optional. When absent, SubagentRunner uses whatever agent is set on the cloned client (the main session's agent). When present, it calls client.set_agent(id) on the clone. The caller can pass any valid Dust agent sId or slug recognized by the Dust workspace.

8 Configuration

MODIFIED

No new TOML configuration keys are required for the initial implementation. The subagent tool inherits the authenticated session's workspace and Dust API URL automatically.

A future enhancement could add a [dust] default_subagent_id key, but for now the tool's agent_id parameter covers that use case.

The maximum subagent nesting depth defaults to 3 (defined as MAX_SUBAGENT_DEPTH in src/mcp/subagent.rs). A future enhancement could expose this as a [mcp] max_subagent_depth TOML key.

Tool visibility

The oxide_agent tool is only registered when a DustClient is available (i.e., the user is logged in). If the user is not authenticated, McpManager::init receives None for the client, and the tool is silently omitted from the tool list — consistent with how the bash tool is only shown when configured.

Auto-approval

oxide_agent is added to the built-in auto-approve list alongside oxide_bash. It is never shown in the interactive approval UI. This is intentional: subagent calls are made by Dust agents as part of automated workflows, and pausing to ask the user for each spawn defeats the purpose of agent orchestration.

The auto-approve list lives in src/mcp/mod.rs (or wherever the existing approval gate is implemented). Add "oxide_agent" to that list. Per-tool configurable auto-approve (letting users whitelist arbitrary tool names in config.toml) is a future extension and is out of scope here.

9 Error Handling and Timeouts

Timeout strategy

The existing wait_for_agent_message already has a 30-second timeout (100 polls × 300ms). For subagent calls, this is the main risk: a stuck subagent will block its tokio::spawn task for up to 30 seconds.

Mitigation: Wrap run_subagent in tokio::time::timeout with a configurable duration (default 120 seconds — generous enough for complex agents).

    const SUBAGENT_TIMEOUT: Duration = Duration::from_secs(120);

    let result = tokio::time::timeout(
        SUBAGENT_TIMEOUT,
        crate::mcp::subagent::run_subagent(&client, prompt, agent_id, depth),
    )
    .await
    .unwrap_or_else(|_| Err(anyhow::anyhow!("subagent timed out after 120s")));

Error propagation

  • Dust API errors (HTTP errors, SSE parse failures) surface as is_error: true tool results with the error message as content. The calling agent can decide how to react.
  • Auth errors (expired token) are handled inside token_refresh::get_valid_token() which auto-refreshes. If that fails, the error propagates up to the tool result.
  • The TUI itself never panics or crashes — the subagent runs in a spawned task.

The description field

The description parameter serves two purposes:

  1. TUI status bar — shown while the subagent is running (e.g., "[subagent: reviewing code...]"), making long-running calls transparent to the user.
  2. Conversation title — used as the title of the Dust conversation created for this subagent call (prefixed with [subagent]), so it shows up clearly when the user runs /resume. If omitted, the first 60 characters of the prompt are used instead.

It does not otherwise affect the Dust API call or the prompt sent to the agent.

14 Risks and Mitigations

Recursive subagent depth

Risk: The Dust agent called as a subagent might itself call oxide_agent, creating arbitrarily deep recursion that exhausts resources or causes a runaway loop.

Mitigation: A configurable max depth (default: 3) is enforced in run_subagent. The current depth is threaded through the call chain via the McpManager::current_depth field. When the depth exceeds the limit, run_subagent returns immediately with a clear error string as the tool result (e.g., "oxide_agent max depth (3) exceeded — cannot spawn further subagents"). The calling agent receives this as a tool error and can decide how to proceed.

Remote subagents — tool calls handled by external MCP servers that Dust has configured server-side in an agent's workspace — are not counted against the local depth limit. Only calls that flow back through oxide's own oxide_agent built-in tool are tracked.

Blocking the McpManager mutex

Risk: call_tool holds the McpManager mutex while awaiting run_subagent. If two subagent calls happen concurrently (e.g., the Dust agent calls oxide_agent twice in parallel), the second call will block waiting for the mutex.

Mitigation (short term): Clone the DustClient out of the manager before any async work, then release the lock immediately. The structure in section 4.3 already does this (.clone() before the await). The run_subagent call itself does not hold the mutex.

Review: the code in section 4.3 must be structured so the mutex is dropped before the run_subagent await. Since call_tool takes &mut self, it holds the mutex for its entire duration — this is a problem for parallel calls.

Correct fix: Extract the client and timeout value before the await, but since &mut self is held across the await, the real solution is to structure call_tool to clone everything needed, then return a future that doesn't hold self. The simplest approach: clone the DustClient synchronously, then call the async runner. The mutex is already released by tokio::spawn in main.rs — the spawned task acquires the lock, runs, and releases it. Parallel tasks queue up but don't deadlock.

15 Subagent Conversations in /resume

Each subagent call creates a real Dust conversation. These conversations should be discoverable via /resume so users can inspect what the subagent did — providing transparency without any extra effort.

Conversation title

When run_subagent creates the conversation via create_conversation_with_skills, it should pass a title that makes the subagent origin clear. The recommended format is:

[subagent] {first 60 chars of prompt}

If the caller also supplies a description parameter, prefer:

[subagent] {description}

This prefix makes subagent conversations immediately identifiable in the /resume picker list, which shows conversation titles.

Implementation

The run_subagent function signature already accepts the prompt and optionally the description. Pass the derived title into create_conversation_with_skills (or the underlying POST /conversations call). No schema changes are needed — the existing conversation create endpoint already accepts a title field. The /resume picker loads conversations from the Dust API and will surface them automatically.

Future extension: per-tool auto-approve

oxide_agent is hard-coded to auto-approve in this implementation (see section 8). A future extension would allow users to configure auto-approve per tool name in config.toml, e.g. [mcp.auto_approve] = ["oxide_agent", "my_custom_tool"]. This is out of scope for the current plan.
