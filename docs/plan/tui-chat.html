Oxide TUI Chat — Implementation Plan

Goal: Build a terminal UI (ratatui) for Oxide that looks and feels like Claude Code / OpenCode — a scrollable message history area on top and an expandable text input at the bottom.
Branch: feat/tui-chat
Scope: UI shell only — no Dust API integration yet. We wire a simple echo/mock agent so the layout is testable end-to-end.

6 Step 4 — Message History Widget

ui/messages.rs
  • Render each message as a block: role label (colored) + content lines
  • User messages: green "you" label, left-aligned
  • Agent messages: orange @agent-name label (from Role::Agent(name)), left-aligned

7 Step 5 — Input Box

input_buffer.rs ui/input.rs
  • Custom InputBuffer tracks text content and byte-offset cursor
  • Rendered via ratatui Paragraph + Wrap { trim: false } for native soft-wrapping
  • Cursor position computed accounting for visual wraps
  • Input box height grows dynamically based on content (clamped to half terminal)
  • Bordered box with placeholder text "Type a message..."
  • Enter → submit message
  • Alt+Enter → newline (multi-line input)
  • Clear input after submit

Why not tui-textarea?

tui-textarea does not support soft line wrapping — long input scrolls horizontally off-screen, which breaks readability. A custom InputBuffer paired with ratatui's Paragraph + Wrap gives us native soft-wrapping at the terminal boundary, dynamic box height, and full control over cursor placement. The trade-off is maintaining basic editing (insert, delete, cursor movement) ourselves, but the surface area is small and well-tested (18 unit tests).

8 Step 6 — Event Handling

event.rs handler.rs
  • event.rs: async task that reads crossterm events and sends them over a tokio::mpsc channel
  • handler.rs: maps every key event to an intent-level Action:
    • Ctrl+C / Ctrl+D → Quit
    • Enter → Submit
    • Alt+Enter → InsertNewline
    • Backspace / Delete → DeleteBack / DeleteForward
    • Arrow keys, Home, End → MoveLeft, MoveRight, MoveHome, MoveEnd
    • PageUp / PageDown → ScrollUp / ScrollDown
    • Printable chars → InsertChar(c)
  • apply_action() in handler.rs applies each Action to App + InputBuffer. main.rs only runs the event loop and render loop.

! Known Limitations

Text wrapping is UTF-8 safe but not display-width aware

All wrapping and cursor logic uses chars().count() (one char = one column). This is safe for ASCII and most multi-byte UTF-8 text. However, double-width characters (CJK ideographs, some emoji) occupy two terminal columns, so wrapping and cursor placement can drift by one column for those characters. Full display-width support would require a crate like unicode-width.
