Oxide Subagent Inline Activity Blocks

Goal: Render subagent calls as inline activity blocks in the messages area, modelled after Claude Code's Agent tool display. While a subagent runs, a spinning indicator appears between the triggering user message and the agent's final reply. On completion the block updates in-place to show duration.
Scope: New Message variant, call-ID correlation, in-place mutation, renderer, spinner animation via tick events. Status bar subagent count kept but made secondary.
Success: User sees ⟳ Agent(description) appear immediately when oxide_agent is invoked, and sees ✓ Agent(description) — 3s when the call completes. No layout regressions in other message types.

1 Current State

Message model — src/app.rs

The Message struct currently has two fields: role: Role and content: String. The Role enum has three variants: User, Agent(String), System. There is no variant for subagent activity; the only trace of subagents in the message layer is App::subagent_count: usize which drives a status bar indicator.

Event model — src/dust/client.rs

DustEvent has two subagent variants already:

SubagentStarted { description: Option<String> }
SubagentFinished { description: Option<String> }

Both fields are #[allow(dead_code)] — the description is never read in main.rs. The current handlers only call app.increment_subagent() / app.decrement_subagent().

Where events are fired — src/mcp/mod.rs

Inside McpManager::call_tool, when tool_name == "oxide_agent", the manager sends SubagentStarted before calling run_subagent_with_timeout and SubagentFinished after. The description value comes from the tool's optional description JSON input field.

There is no unique identifier sent with these events. Matching start to finish currently relies on the description string — which is fragile when two concurrent subagents share the same description.

Renderer — src/ui/messages.rs

render_messages iterates app.messages() and produces Vec<Line>. Tool approval blocks are appended at the end via tool_approval_lines() only when AppMode::ToolApproval is active. There is no inline block concept yet.

Tick events — src/event.rs

AppEvent::Tick fires every 250 ms when no terminal event is pending. The main loop has Some(AppEvent::Tick) => {} — it does nothing today. This is the hook we will use to animate the spinner.

2 Design Decisions

Decision A — New Role variant, not a separate struct

Add Role::SubagentCall(SubagentCallState) so subagent blocks sit in the existing Vec<Message> alongside user/agent/system messages. This preserves scroll logic and all existing rendering loops without a parallel data structure. The renderer pattern-matches on role just as it does today for Role::User / Role::Agent.

Decision B — UUID call ID for correlation

Add a call_id: String field to both SubagentStarted and SubagentFinished events. The MCP manager generates this ID (UUID v4 via the uuid crate, already in Cargo.toml, or a simple incrementing u64 if not). app.rs uses the ID to find and mutate the correct SubagentCall message. This handles concurrent subagent calls safely.

Decision C — In-place mutation via call_id lookup

On SubagentStarted, push a new Message { role: Role::SubagentCall(...), content: "" } and record its position by scanning for the matching call_id. On SubagentFinished, iterate messages in reverse to find the SubagentCall with matching call_id and update its state to Done with elapsed duration. No deduplication needed — each call_id is unique.

Decision D — Spinner via tick counter in App

Add tick: usize to App. On each AppEvent::Tick, call app.tick() which increments the counter. The renderer reads app.tick() to select a spinner frame from a small static array ["⟳", "↻"] or a Unicode braille spinner. This is pure render-time; no re-allocation on tick.

Decision E — Status bar count kept, not removed

The status bar text [N subagent(s) running] is kept as-is. It provides a quick numeric summary that remains visible even when the user has scrolled up away from the inline blocks. The inline block is the primary UX; the status bar is the secondary fallback. No code needs to be removed from layout.rs.

Updated Role enum — src/app.rs

#[derive(Debug, Clone, PartialEq, Eq)]
pub enum Role {
    User,
    Agent(String),
    System,
    SubagentCall(SubagentCallState),  // NEW
}

Note: SubagentCallState contains std::time::Instant which does not implement PartialEq or Eq. The PartialEq derive on Role will need a manual implementation or the derive must be replaced. The simplest fix: implement PartialEq manually for Role, comparing only the discriminant for the SubagentCall arm (tests that compare roles use assert!(matches!(...)) style, not ==, except for one test that compares Role::User and Role::Agent). Alternatively, wrap started_at in a newtype that implements PartialEq as always-equal, or store start time as u64 unix timestamp instead of Instant.

Recommended approach: Store started_at as std::time::Instant (for accuracy) but implement PartialEq manually for Role, making SubagentCall(_) == SubagentCall(_) always false (two subagent calls are never semantically equal). The existing tests that use assert_eq!(msgs[0].role, Role::User) will still compile because they only test User/Agent/System.

New App methods — src/app.rs

/// Push a new Running subagent block into the message list.
pub fn push_subagent_started(&mut self, call_id: String, description: Option<String>) {
    self.messages.push(Message {
        role: Role::SubagentCall(SubagentCallState {
            call_id,
            description,
            status: SubagentCallStatus::Running,
            started_at: std::time::Instant::now(),
        }),
        content: String::new(),
    });
    self.scroll_offset = 0;
    self.subagent_count += 1;
}

/// Find the SubagentCall with matching call_id and mark it Done/Failed.
pub fn complete_subagent(&mut self, call_id: &str, success: bool) {
    for msg in self.messages.iter_mut().rev() {
        if let Role::SubagentCall(state) = &mut msg.role {
            if state.call_id == call_id {
                let elapsed = state.started_at.elapsed().as_secs();
                state.status = if success {
                    SubagentCallStatus::Done { elapsed_secs: elapsed }
                } else {
                    SubagentCallStatus::Failed { elapsed_secs: elapsed }
                };
                break;
            }
        }
    }
    self.subagent_count = self.subagent_count.saturating_sub(1);
}

/// Increment tick counter (called on every AppEvent::Tick).
pub fn tick(&mut self) {
    self.tick = self.tick.wrapping_add(1);
}

pub const fn tick_count(&self) -> usize {
    self.tick
}

The existing increment_subagent / decrement_subagent methods are removed; their callers in main.rs are replaced.

4 Event Flow Changes

oxide_agent tool call invoked in McpManager::call_tool | | Generate call_id = uuid::Uuid::new_v4().to_string() | (or: atomic u64 counter stored in McpManager) | v tx.send(DustEvent::SubagentStarted { call_id: call_id.clone(), description: ... }) | v run_subagent_with_timeout(...).await ← blocks async task | v tx.send(DustEvent::SubagentFinished { call_id, description, success }) ------- main.rs event loop (dust_rx branch) ------- DustEvent::SubagentStarted { call_id, description } v app.push_subagent_started(call_id, description) → pushes Message { role: SubagentCall(Running), content: "" } → resets scroll to bottom → increments subagent_count DustEvent::SubagentFinished { call_id, success, .. } v app.complete_subagent(&call_id, success) → finds SubagentCall by call_id in messages (reverse scan) → mutates status → Done { elapsed_secs } or Failed { elapsed_secs } → decrements subagent_count AppEvent::Tick v app.tick() ← was a no-op, now increments counter → renderer reads tick_count() to animate spinner

Concurrency note

Multiple concurrent oxide_agent calls are possible (the parent agent can call the tool multiple times in a single turn). Each call runs in its own tokio::spawn task inside McpManager::call_tool. The event_tx channel is UnboundedSender, which is Send + Clone, so multiple tasks can send on it concurrently without mutex. The call_id ensures each SubagentFinished matches exactly one SubagentStarted block.

5 Call ID Generation

Two options exist. The simplest that avoids a new dependency is an atomic counter on McpManager:

// In McpManager struct:
next_call_id: std::sync::atomic::AtomicU64,

// Usage:
let call_id = self.next_call_id
    .fetch_add(1, std::sync::atomic::Ordering::Relaxed)
    .to_string();

This is monotonically increasing within a session, unique, and requires no external crate. A UUID would be more robust across serialisation boundaries but oxide's event channel is in-process only, so the counter is sufficient.

Recommendation: Use AtomicU64 counter. Add next_call_id: std::sync::atomic::AtomicU64 to McpManager, initialised to AtomicU64::new(0). The call_id in events is a String (e.g. "0", "1", …) so the type is stable and no crate changes are needed.

Spinner frames

const SPINNER_FRAMES: &[&str] = &["⟳", "↻", "⟳", "↻"];
// or a braille spinner for smoother feel:
const SPINNER_FRAMES: &[&str] = &["⣾", "⣽", "⣻", "⢿", "⡿", "⣟", "⣯", "⣷"];

Tick fires every 250 ms. With 8 braille frames the spinner cycles every 2 s. With the simple 2-frame version it alternates every 250 ms. Either works; the 2-frame version is safe on all terminals.

Rendering logic in src/ui/messages.rs

Add a new branch inside the for msg in app.messages() loop:

Role::SubagentCall(state) => {
    lines.push(Line::from(""));
    lines.extend(subagent_call_lines(state, app.tick_count()));
}
Role::User => { ... }
Role::Agent(_) => { ... }

Implement a free function:

fn subagent_call_lines(
    state: &SubagentCallState,
    tick: usize,
) -> Vec<Line<'static>> {
    let label = state.description.as_deref().unwrap_or("subagent");

    match &state.status {
        SubagentCallStatus::Running => {
            let frame_idx = tick % SPINNER_FRAMES.len();
            let spinner = SPINNER_FRAMES[frame_idx];
            vec![Line::from(vec![
                Span::styled(
                    format!("  {spinner} "),
                    Style::default().fg(Color::Yellow),
                ),
                Span::styled(
                    format!("Agent({label})"),
                    Style::default()
                        .fg(Color::Yellow)
                        .add_modifier(Modifier::BOLD),
                ),
            ])]
        }
        SubagentCallStatus::Done { elapsed_secs } => {
            vec![Line::from(vec![
                Span::styled(
                    "  ✓ ",
                    Style::default().fg(Color::Green),
                ),
                Span::styled(
                    format!("Agent({label})"),
                    Style::default()
                        .fg(Color::Green)
                        .add_modifier(Modifier::BOLD),
                ),
                Span::styled(
                    format!("  ({elapsed_secs}s)"),
                    Style::default().fg(Color::DarkGray),
                ),
            ])]
        }
        SubagentCallStatus::Failed { elapsed_secs } => {
            vec![Line::from(vec![
                Span::styled(
                    "  ✗ ",
                    Style::default().fg(Color::Red),
                ),
                Span::styled(
                    format!("Agent({label})"),
                    Style::default()
                        .fg(Color::Red)
                        .add_modifier(Modifier::BOLD),
                ),
                Span::styled(
                    format!("  failed ({elapsed_secs}s)"),
                    Style::default().fg(Color::DarkGray),
                ),
            ])]
        }
    }
}

The function signature takes &SubagentCallState and usize tick — both cheaply copyable, no lifetime issues, testable in isolation with TestBackend.

PartialEq on Role — test impact

The render_messages function only reads msg.role via pattern match — it does not use ==. The test suite in app.rs uses assert_eq!(msgs[0].role, Role::User) for User/Agent/System but never for SubagentCall. Adding a manual PartialEq implementation that covers all four arms will keep existing tests green. The manual impl should compare SubagentCall arms by call_id for correctness.

8 PartialEq Manual Implementation

The existing #[derive(PartialEq, Eq)] on Role must be replaced because SubagentCallState contains std::time::Instant which does not implement PartialEq. Replace the derive with:

impl PartialEq for Role {
    fn eq(&self, other: &Self) -> bool {
        match (self, other) {
            (Role::User, Role::User) => true,
            (Role::Agent(a), Role::Agent(b)) => a == b,
            (Role::System, Role::System) => true,
            // Two SubagentCall roles compare equal only when their call_ids match.
            // In practice the test suite never compares two SubagentCall roles.
            (Role::SubagentCall(a), Role::SubagentCall(b)) => a.call_id == b.call_id,
            _ => false,
        }
    }
}

impl Eq for Role {}

All existing tests in app.rs that assert msgs[0].role == Role::User continue to pass unchanged.

9 Spinner Animation Details

Tick fires every 250 ms (the EventReader poll timeout). The App::tick counter increments on every tick event. The renderer reads tick % SPINNER_FRAMES.len() to select the frame.

  • With 2 frames: alternates every 250 ms (4 Hz)
  • With 8 braille frames: cycles every 2 s (0.5 Hz per frame)
  • Recommended: use the 2-frame ["⟳", "↻"] for broad terminal compatibility. The braille dots can render as squares on some older terminals / fonts.

The tick counter wraps via wrapping_add(1) — no overflow risk.

The spinner only animates when a SubagentCall message with status Running exists. The frame is selected at render time per-frame; there is no stored "current frame" state. This is stateless from the renderer's perspective.

12 Risks and Mitigations

Risk: Instant in Role breaks PartialEq derive

Mitigation: Manual PartialEq impl as described in Section 8. All existing tests continue to compile and pass.

Risk: SubagentFinished arrives before SubagentStarted is processed

Impossible in practice. Events are sent sequentially on the same UnboundedSender from the same async task (SubagentStarted then await then SubagentFinished). The channel preserves ordering. The main loop processes events one at a time.

Risk: Concurrent subagents with same description cause confusion

Mitigation: The call_id (not description) is used for all lookup/mutation. Descriptions are display-only.

Risk: Very long description overflows terminal width

The subagent label is rendered as a single Span. If the description is longer than the terminal width, Ratatui will truncate it (Paragraph behaviour). The renderer should clip or truncate the label to body_width - 10 chars to avoid wrapping onto a second line. Add a truncate_label(label, max) helper that appends … if needed.

Risk: Elapsed time is 0s for very fast subagents

Acceptable — display as (0s) or (<1s). Consider showing milliseconds when elapsed < 1s: elapsed_millis: u64 stored alongside elapsed_secs, or derive from Instant at render time. Simplest: store started_at: Instant in state and compute elapsed at render time instead of at complete_subagent call. This avoids storing a redundant field and gives sub-second resolution for display.

13 Revised Data Model (incorporating risk mitigations)

Store started_at: std::time::Instant in SubagentCallState and compute elapsed at render time. The SubagentCallStatus enum needs no elapsed field:

#[derive(Debug, Clone)]
pub enum SubagentCallStatus {
    Running,
    Done,
    Failed,
}

#[derive(Debug, Clone)]
pub struct SubagentCallState {
    pub call_id: String,
    pub description: Option<String>,
    pub status: SubagentCallStatus,
    pub started_at: std::time::Instant,
    pub finished_at: Option<std::time::Instant>,
}

In complete_subagent, set state.finished_at = Some(std::time::Instant::now()). In the renderer, compute elapsed = state.finished_at.unwrap().duration_since(state.started_at) and format as <1s (when < 1000 ms) or Xs. This approach also avoids storing duplicate data and gives sub-second accuracy.
