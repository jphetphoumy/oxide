Oxide — Animated Streaming Indicator in Conversation

Goal: Render an animated "Thinking..." block with spinner and elapsed time directly inside the message area while the agent is streaming, immediately after the agent's response content.
Scope: App state change (streaming_started_at), new rendering function in messages.rs, integration into the message rendering loop. No Dust API changes needed.
Success: A yellow spinner + "Thinking... Xs" line appears below the last agent message header while streaming, updates every tick (250ms), and disappears when complete_stream() is called.

1 Current State & Context

What already exists

  • app.rs:101 — is_streaming: bool tracks whether the agent SSE stream is active.
  • app.rs:107 — tick: u64 wraps every frame (250ms tick interval).
  • messages.rs:10 — SPINNER_FRAMES: &[&str] = &["⟳", "↻"] already declared module-level.
  • messages.rs:203-260 — subagent_call_lines() renders a spinner + elapsed time for running subagent blocks. This is the exact pattern to replicate.
  • layout.rs:58-59 — Status line shows " streaming..." text when streaming. This can be removed or kept (see design decision below).
  • main.rs:908-910 — AppEvent::Tick calls app.tick() every 250ms, triggering redraws.

Key pattern to replicate: subagent_call_lines()

The subagent block computes elapsed time as started_at.elapsed().as_secs() for running state. For streaming, we need the same pattern: store a std::time::Instant when streaming starts, compute elapsed().as_secs() at render time.

The subagent spinner uses: (tick as usize) % SPINNER_FRAMES.len() — identical approach will work for the streaming indicator.

Design decision: status line vs. inline indicator

The status line at layout.rs:58 already shows " streaming...". With this feature, the inline indicator becomes the primary visual feedback. The status line text should either be removed (clean) or kept as redundant secondary info. The plan recommends removing the streaming text from the status line since the inline indicator provides richer information (spinner + elapsed time) at the right visual position.

2 Architecture

Data flow with the new feature

user sends message | v app.send_message() ← sets is_streaming = true, streaming_started_at = Some(Instant::now()) | v AppEvent::Tick ← fires every 250ms, increments app.tick | v render_frame() ← calls render_messages(frame, app, layout.messages) | v render_messages() ← after rendering the last Agent message, if app.is_streaming() | inserts streaming_indicator_lines(app.streaming_started_at(), tick) v ⟳ Thinking... 3s ← rendered inline below agent label/content | v DustEvent::Complete ← calls app.complete_stream() | sets is_streaming = false, streaming_started_at = None v indicator disappears, final message content shown

Rendering placement decision

The indicator must appear after the last Agent message's content lines, not as a separate message entry. This means the injection point is inside the Role::Agent match arm in render_messages(), conditioned on app.is_streaming() AND this being the last message.

Alternative: append indicator lines outside the message loop, unconditionally when is_streaming. This is simpler and equally correct because send_message() always pushes an Agent placeholder as the last message when streaming starts.

Chosen approach: append the streaming indicator block after the main messages loop, gated on app.is_streaming(). This avoids complicating the per-message match arms and mirrors how ToolApproval lines are appended after the loop.

3.2 — src/ui/messages.rs

New function: streaming_indicator_lines()

fn streaming_indicator_lines(
    started_at: Option<std::time::Instant>,
    tick: u64,
) -> Vec<Line<'static>> {
    #[allow(clippy::cast_possible_truncation)]
    let frame_idx = (tick as usize) % SPINNER_FRAMES.len();
    let spinner = SPINNER_FRAMES[frame_idx];

    let elapsed_secs = started_at
        .map(|t| t.elapsed().as_secs())
        .unwrap_or(0);

    vec![
        Line::from(""),
        Line::from(vec![
            Span::styled(
                format!("  {spinner} "),
                Style::default().fg(Color::Yellow),
            ),
            Span::styled(
                "Thinking...",
                Style::default()
                    .fg(Color::Yellow)
                    .add_modifier(Modifier::BOLD),
            ),
            Span::styled(
                format!("  {elapsed_secs}s"),
                Style::default().fg(Color::DarkGray),
            ),
        ]),
    ]
}

    let mut lines: Vec<Line> = Vec::new();
    let body_width = (area.width as usize).saturating_sub(INDENT.len());

    for msg in app.messages() {
        match &msg.role {
            Role::User => { /* ... */ }
            Role::Agent(name) => { /* ... */ }
            Role::System => { /* ... */ }

    }

        lines.extend(tool_approval_lines(/* ... */));
    }

    // ... scroll + widget render ...
}

6 Edge Cases & Interaction Notes

Simultaneous streaming + subagents

Subagent blocks are already rendered as SubagentCall messages in the messages loop. The streaming indicator is appended after all messages. If subagents are running while the main agent is still streaming, both will be visible:

  ⟳ Agent(fetch data)    ← subagent block (inside loop)
  ⟳ Thinking...  5s      ← streaming indicator (after loop)

This is intentional — they represent different things. No special handling required.

Streaming during ToolApproval mode

When AppMode::ToolApproval is active, is_streaming may still be true (the stream paused waiting for tool result). In this case both the streaming indicator and the tool approval block will render. The streaming indicator appears above the tool approval lines, which is acceptable — the tool approval block takes visual priority as it requires user action. No change needed.

Scroll behavior

The streaming indicator adds lines to the total line count. Since scroll_offset is reset to 0 on send_message(), the user starts at the bottom and sees the indicator. If the user scrolls up during streaming, the indicator remains at the logical bottom and is not visible — this is correct behavior matching how existing content works.

new_conversation() and restore_conversation()

Both call push_system_message() or directly set is_streaming = false. The new_conversation() path calls push_system_message() which will clear streaming_started_at. The restore_conversation() path directly sets self.is_streaming = false — add self.streaming_started_at = None; there as well.

Elapsed time precision

The elapsed time uses .as_secs() (integer seconds). Sub-second display (e.g., "0.3s") is not needed — the subagent block uses the same integer-second approach and it is well-received. Keeping it consistent avoids extra complexity.

8 Open Questions

Q1: Keep or remove status line "streaming..." text?

Recommendation: Remove it. The inline indicator provides better UX (positioned near the content, shows elapsed time, animated spinner). The status line can use that space for other info. If desired, a minimal "S" or dot could remain as a redundant indicator — but this is unnecessary given the inline block.

Q2: Should "Thinking..." be customizable?

The requirements ask about "Processing...", "Executing..." alternatives. For now, hard-code "Thinking..." as a constant: const STREAMING_LABEL: &str = "Thinking...";. Future customization (e.g., per-agent label or config option) can be added later. No config or API change is needed for this iteration.

Q3: What happens if streaming never started_at is None but is_streaming is true?

This should not happen with the proposed implementation since both are set together in send_message(). The streaming_indicator_lines() function uses .unwrap_or(0) for elapsed, so if started_at is None, it shows "0s" gracefully. No panic risk.
