Mouse Scroll for Message History

Enable terminal-native mouse wheel scrolling through conversation history, matching the UX of Claude Code and Codex.

Problem

When an AI agent returns a long response, the user cannot scroll back up to review earlier parts of the conversation. The only scroll mechanism today is PageUp/PageDown (5 lines at a time), which is not discoverable and doesn't match the expected UX of modern terminal tools like Claude Code or Codex where the mouse wheel just works.

There are two underlying issues:

  1. bug Scroll direction inverted in renderer — scroll_offset = 0 is treated as "at the bottom" in app.rs (resets to 0 on new messages, was_at_bottom = offset == 0), but the renderer passes it directly to Paragraph::scroll((offset, 0)), where 0 means "show from the top". The semantics disagree.
  2. feat No mouse wheel handling — event.rs only forwards Event::Key events to the app loop. Mouse wheel events (Event::Mouse with MouseEventKind::ScrollUp/ScrollDown) are silently discarded despite EnableMouseCapture being active.

Implementation Steps

Step 1: Fix scroll direction bug

In src/ui/messages.rs, the render function must invert the offset so that scroll_offset = 0 maps to Paragraph::scroll((max_scroll, 0)) (bottom of content).

// Before (broken):
let scroll = app.scroll_offset().min(max_scroll);

// After (fixed):
let clamped = app.scroll_offset().min(max_scroll);
let scroll = max_scroll.saturating_sub(clamped);

This fix is independent of mouse support and should be committed separately. It also fixes the existing PageUp/PageDown behavior which is currently reversed.

Step 2: Forward mouse events from the event reader

In src/event.rs, add a Mouse variant to AppEvent and forward Event::Mouse from the crossterm poll loop:

pub enum AppEvent {
    Key(crossterm::event::KeyEvent),
    Mouse(crossterm::event::MouseEvent),
    Tick,
}

// In the poll loop, handle both Key and Mouse:
match event::read() {
    Ok(Event::Key(key)) if key.kind == KeyEventKind::Press => {
        if tx.send(AppEvent::Key(key)).is_err() { return; }
    }
    Ok(Event::Mouse(mouse)) => {
        if tx.send(AppEvent::Mouse(mouse)).is_err() { return; }
    }
    _ => {}
}

Step 3: Map mouse scroll to existing scroll actions

In src/handler.rs, add a function that converts mouse scroll events to Action:

use crossterm::event::{MouseEvent, MouseEventKind};

const MOUSE_SCROLL_LINES: usize = 3;

pub fn handle_mouse_event(mouse: MouseEvent) -> Action {
    match mouse.kind {
        MouseEventKind::ScrollUp => Action::ScrollUp(MOUSE_SCROLL_LINES),
        MouseEventKind::ScrollDown => Action::ScrollDown(MOUSE_SCROLL_LINES),
        _ => Action::None,
    }
}

3 lines per scroll tick is a common default. This can be tuned later. The key point is reusing the existing Action::ScrollUp/ScrollDown variants — no new action types needed.

Step 4: Wire mouse events into the main loop

In src/main.rs, add a match arm for AppEvent::Mouse alongside the existing AppEvent::Key handler:

Some(AppEvent::Mouse(mouse)) => {
    if matches!(app.mode(), AppMode::Chat) {
        let action = handle_mouse_event(mouse);
        apply_action(&mut app, &mut input, action);
    }
}

Mouse scroll is only processed in Chat mode — in Picker mode, the existing Up/Down keys handle navigation.

Step 5: Update tests

  • src/ui/messages.rs — no render tests to update (the scroll direction fix changes behavior but existing unit tests only cover wrap_line)
  • src/handler.rs — add tests for handle_mouse_event(): ScrollUp → Action::ScrollUp(3), ScrollDown → Action::ScrollDown(3), other mouse events → Action::None
  • src/event.rs — no unit tests (async tokio spawn, tested via integration)
