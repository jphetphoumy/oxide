Oxide Login / Logout — Implementation Plan

Goal: Implement oxide login and oxide logout using the same WorkOS OAuth device code flow as the official Dust CLI. Tokens are stored in the OS keyring and refreshed automatically.
Branch: feat/auth
Scope: CLI subcommands (login, logout, status), OAuth device flow, token storage, token refresh. No API calls to Dust yet.

0 Login Flow Overview

oxide login | v POST https://api.workos.com/user_management/authorize/device | body: client_id + scope v Response: device_code, user_code, verification_uri_complete, interval | v Print to terminal: Your code: ABCD-1234 Opening browser to verify... | | open(verification_uri_complete) in default browser v Poll every {interval}s: POST https://api.workos.com/user_management/authenticate | body: grant_type=device_code, device_code, client_id | | "authorization_pending" -> keep polling | "slow_down" -> poll interval += 5s | success -> got tokens! v Tokens received: access_token (JWT), refresh_token | | 1. Decode JWT -> extract region from "https://dust.tt/region" claim | 2. Store access_token, refresh_token, region in OS keyring | 3. Workspace selection (fetch workspaces, user picks one, store ID) v Done: "Logged in as user@example.com (workspace: Acme Corp)"

1 WorkOS Endpoints

1. Initiate device code flow

POST https://api.workos.com/user_management/authorize/device 
Content-Type: application/x-www-form-urlencoded

client_id=client_01JGCT55T7FVDG9XF74925R1KT
&scope=openid profile email

Response

{
  "device_code": "...",
  "user_code": "ABCD-1234",
  "verification_uri": "https://auth.workos.com/device",
  "verification_uri_complete": "https://auth.workos.com/device?user_code=ABCD-1234",
  "expires_in": 600,
  "interval": 5
}

2. Poll for token exchange

POST https://api.workos.com/user_management/authenticate 
Content-Type: application/x-www-form-urlencoded

grant_type=urn:ietf:params:oauth:grant-type:device_code
&device_code={device_code}
&client_id=client_01JGCT55T7FVDG9XF74925R1KT

Polling responses

ResponseAction
{"error": "authorization_pending"}Keep polling at same interval
{"error": "slow_down"}Add 5 seconds to poll interval
{"error": "expired_token"}Device code expired — restart flow
{"access_token": "...", "refresh_token": "..."}Success — store tokens

3. Refresh tokens

POST https://api.workos.com/user_management/authenticate 
Content-Type: application/x-www-form-urlencoded

grant_type=refresh_token
&client_id=client_01JGCT55T7FVDG9XF74925R1KT
&refresh_token={stored_refresh_token}

On 400/401 response: clear stored tokens, user must oxide login again.

/// Returns an error if no tokens are stored (user must `oxide login`).
pub async fn get_valid_token() -> Result<String> {
    let access_token = token_storage::get_access_token()?
        .ok_or_else(|| anyhow!("Not logged in. Run `oxide login` first."))?;

    if !is_token_expired(&access_token) {
        return Ok(access_token);
    }

    // Token expired or close to expiry — refresh it
    refresh_tokens().await
}

src/main.rs

pub fn logout() -> Result<()> {
    token_storage::clear_all()?;
    println!("Logged out. Credentials cleared.");
    Ok(())
}

oxide status

pub async fn status() -> Result<()> {
    match token_storage::get_access_token()? {
        None => println!("Not logged in. Run `oxide login`."),
        Some(token) => {
            let expired = token_refresh::is_token_expired(&token);
            let region = token_storage::get_region()?.unwrap_or_default();
            let workspace = token_storage::get_workspace_id()?.unwrap_or_default();
            println!("Logged in");
            println!("  Region:    {}", region);
            println!("  Workspace: {}", workspace);
            println!("  Token:     {}", if expired { "expired" } else { "valid" });
        }
    }
    Ok(())
}

11 Implementation Order

#MilestoneDeliverable
1 CLI subcommands clap with login, logout, status. No subcommand → TUI. Compiles and runs.
2 Token storage token_storage.rs — save/get/clear for all 4 keys. Tested manually with oxide status after writing a dummy value.
3 Device flow oxide login completes the full WorkOS device code flow. Tokens + region stored in keyring. Browser opens automatically.
4 Token refresh get_valid_token() checks expiry and refreshes if needed. Tested by waiting for token to near-expire then calling status.
5 Logout & status oxide logout clears keyring. oxide status shows login state, region, workspace, token validity.

12 Testing Strategy

  • device_flow.rs — extract_region() with various JWTs (valid claim, missing claim, malformed)
  • token_refresh.rs — is_token_expired() with expired, valid, and near-expiry JWTs
  • device_flow.rs — TokenPollResponse deserialization for success and error variants

Integration tests (manual)

  • oxide login — full flow: browser opens, code displayed, tokens stored after approval
  • oxide status — shows correct state before and after login
  • oxide logout — clears credentials, status shows "not logged in"
  • oxide login when already logged in — should re-authenticate (or skip if valid)
  • Token refresh — wait for JWT to near-expire, verify auto-refresh works
