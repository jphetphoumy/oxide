Oxide Context Window Status Bar

Goal: Display the Dust conversation context window usage (used tokens / max tokens) in the status bar of the TUI, so users know how full the context is before hitting limits.
Scope: Polling the Dust context-usage API after each completed agent message, storing the result in App, and rendering a compact indicator in the status bar.
Success: Status bar shows something like ctx: 42% (or a bar) that updates after each response and disappears when there is no active conversation.

1 Feature Overview

The Dust web app already tracks context window usage per conversation by polling GET /api/w/:wId/assistant/conversations/:cId/context-usage. The response returns:

{
  "model": { "providerId": "anthropic", "modelId": "claude-3-7-sonnet-20250219" },
  "contextUsage": 12345,     // promptTokens from the last agent run
  "contextSize": 200000      // max context window of the model
}

Oxide should call this endpoint after every completed agent message and surface the result as a percentage in the status bar. When no conversation is active, or when the data is not yet available (null values), the indicator is hidden.

Why this matters

  • Users working with long conversations in the terminal have no way to know they are approaching the context limit.
  • The web app shows warnings at 70% and 80% — oxide should surface at least the raw percentage so users can self-manage.
  • The Dust API already computes the value server-side; oxide only needs to fetch and display it.

2 Dust API Contract

Endpoint

GET /api/w/{workspaceId}/assistant/conversations/{conversationId}/context-usage

Authentication: bearer token (same pattern as all other Dust API calls). No request body or query parameters.

Response shape

// All fields may be null while a run is still processing
{
  "model": { "providerId": string, "modelId": string } | null,
  "contextUsage": number | null,   // promptTokens used
  "contextSize":  number | null    // model's context window size
}

Behaviour notes (from Dust backend source)

  • Returns all-null when no completed agent run exists yet (new conversation, still processing).
  • Returns the latest stable value while a new run is in-flight (intentional "sticky" design).
  • After a compaction run, uses completionTokens instead of promptTokens as a best-guess estimate.
  • Mounted at /api/w/:wId/... — note this is not the /api/v1/... versioned path.

URL path difference

The existing list_conversations call in client.rs already uses /api/w/{workspace} (not /api/v1/w/{workspace}), so this endpoint follows the same pattern and the existing self.url() helper works fine.

3 Architecture

The implementation follows the established oxide data-flow pattern:

DustEvent::Complete arrives (agent done streaming) | v main.rs handles_dust_message → calls app.complete_stream() AND spawns tokio task: client.fetch_context_usage(conversation_id) | v client.rs GET /api/w/.../context-usage | v DustEvent::ContextUsage { used, size } sent on dust_tx | v main.rs handle_dust_message → app.set_context_usage(used, size) | v ui/layout.rs render_layout reads app.context_usage_percent() and appends ctx: 42% span to status bar

Trigger timing

The fetch is triggered exactly once per completed agent turn, inside handle_dust_message when the DustEvent::Complete variant matches the active conversation ID. This means:

  • No polling loop — one HTTP call per assistant response.
  • The indicator updates after the streaming is done (never mid-stream).
  • If the fetch fails it is silently ignored (non-critical path).

State cleared on new conversation

App::new_conversation() already resets many fields. context_usage must also be cleared there so stale data from a previous conversation is never shown.

src/dust/types.rs

Add a new response DTO for deserializing the context-usage API response:

#[derive(Debug, Clone, PartialEq, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct ContextUsageResponse {
    // All fields are null while the run is still processing
    pub context_usage: Option,
    pub context_size: Option,
    // model field is present but oxide does not need it yet
}

The model field is omitted for now since oxide does not display model information in the status bar. Serde's default behaviour will silently ignore unknown fields.

src/dust/client.rs

New DustEvent variant

pub enum DustEvent {
    // ... existing variants ...
    ContextUsage {
        used: u32,
        size: u32,
    },
}

New method on DustClient

pub async fn fetch_context_usage(
    &self,
    conversation_id: &str,
) -> Result<Option<ContextUsageResponse>> {
    let token = token_refresh::get_valid_token().await?;
    let url = self.url(&format!(
        "/api/w/{}/assistant/conversations/{conversation_id}/context-usage",
        self.workspace_id
    ));
    let response = self
        .http
        .get(&url)
        .bearer_auth(token)
        .send()
        .await
        .context("failed to fetch context usage")?;

    if !response.status().is_success() {
        // Non-critical: silently return None on failure
        return Ok(None);
    }

    let body: ContextUsageResponse = response
        .json()
        .await
        .context("failed to decode context usage response")?;

    Ok(Some(body))
}

Uses self.url() (not self.sse_url()) which already produces the /api/w/{workspace}/... path — matching the list_conversations pattern already in the codebase.

src/app.rs

New field on App

pub struct App {
    // ... existing fields ...
    context_usage: Option<(u32, u32)>, // (used_tokens, total_tokens)
}

New methods

pub fn set_context_usage(&mut self, used: u32, size: u32) {
    if size > 0 {
        self.context_usage = Some((used, size));
    }
}

pub fn context_usage_percent(&self) -> Option<u8> {
    self.context_usage.map(|(used, size)| {
        ((used as f64 / size as f64) * 100.0).round() as u8
    })
}

pub fn clear_context_usage(&mut self) {
    self.context_usage = None;
}

Clear in new_conversation()

pub fn new_conversation(&mut self) {
    self.messages.clear();
    self.is_streaming = false;
    self.conversation_id = None;
    self.scroll_offset = 0;
    self.clear_active_skills();
    self.context_usage = None;  // ← add this line
    self.push_system_message(&format!(
        "Started a new conversation with {}",
        self.agent_name
    ));
}

Also clear in restore_conversation() so switching to a resumed conversation doesn't show stale context from the previous session until the next API call updates it.

src/main.rs

Spawn fetch task on Complete event

Inside handle_dust_message(), in the DustEvent::Complete arm, after calling app.complete_stream(), spawn a task to fetch context usage:

DustEvent::Complete(content, conv_id)
    if conv_id == app.conversation_id().map(ToString::to_string) =>
{
    app.complete_stream(content.as_deref());

    // Fetch context usage asynchronously; result arrives via DustEvent::ContextUsage
    if let (Some(cid), Some(c)) = (
        app.conversation_id().map(ToString::to_string),
        client,
    ) {
        let client_clone = c.clone();
        let dust_tx_clone = dust_tx.clone();
        tokio::spawn(async move {
            match client_clone.fetch_context_usage(&cid).await {
                Ok(Some(usage)) => {
                    if let (Some(used), Some(size)) =
                        (usage.context_usage, usage.context_size)
                    {
                        let _ = dust_tx_clone.send(DustEvent::ContextUsage { used, size });
                    }
                }
                Ok(None) | Err(_) => {
                    // Non-critical; silently ignore
                }
            }
        });
    }
}

Handle new event variant

DustEvent::ContextUsage { used, size } => {
    app.set_context_usage(used, size);
}

src/ui/layout.rs

Add context usage span to status bar

In render_layout(), after the existing spans for agent, cwd, skills, and subagents — add a context usage span. The indicator should appear only when app.context_usage_percent() returns Some.

// Format context usage indicator
let ctx_text = app.context_usage_percent().map_or_else(
    String::new,
    |pct| format!(" ctx:{pct}%"),
);

// Include ctx_text in hint_width calculation (alongside agent_text, cwd_text, etc.)
let hint_width = agent_text.chars().count()
    + cwd_separator.chars().count()
    + cwd_text.chars().count()
    + skills_text.chars().count()
    + subagent_text.chars().count()
    + ctx_text.chars().count()      // ← add this
    + hints.chars().count();

// And in the padding calculation's subtraction:
let cwd_room = usize::from(area.width).saturating_sub(
    agent_text.chars().count()
        + skills_text.chars().count()
        + subagent_text.chars().count()
        + ctx_text.chars().count()  // ← add this
        + hints.chars().count()
        + 2,
);

// Add span after subagent_text span:
if !ctx_text.is_empty() {
    let ctx_color = match app.context_usage_percent().unwrap_or(0) {
        80.. => Color::Red,
        70.. => Color::Yellow,
        _    => Color::DarkGray,
    };
    spans.push(Span::styled(ctx_text, Style::default().fg(ctx_color)));
}

Color coding mirrors the Dust web thresholds: DarkGray below 70%, Yellow at 70-79%, Red at 80%+. This gives users a visual cue consistent with the web UI.

5 Status Bar Layout

The status bar currently renders as a single row of left-to-right spans:

 agent: dust  ~/projects/oxide  [skills: code-review]  Ctrl+C quit  Enter send  Alt+Enter newline

After this feature, the context usage indicator appears between subagents and the right-aligned hints:

 agent: dust  ~/projects/oxide  [skills: code-review]  ctx:42%  Ctrl+C quit  Enter send  Alt+Enter newline

At 70%+ the percentage switches to yellow; at 80%+ it turns red:

 agent: dust  ~/projects/oxide                         ctx:82%  Ctrl+C quit  Enter send  Alt+Enter newline
                                                      ^^^^^^ Red

When there is no active conversation or data is not yet available, the ctx: segment is absent entirely — no layout shift when it appears.

Width accounting

The existing code carefully tracks character widths to right-align the keybinding hints. The ctx_text must be included in both the cwd_room subtraction and the hint_width sum. If this is omitted the hints will overlap or leave extra padding.

8 Edge Cases and Risks

Stale data during streaming

The context usage is fetched after the stream completes, not before. The indicator will show the value from the previous turn while the agent is generating a response. This is acceptable and mirrors the Dust web app's "sticky" behavior.

null response (run still processing)

If the context-usage API returns all nulls (run in flight), fetch_context_usage returns Ok(None) and no event is sent. The indicator keeps the previous value. This is by design.

Network failure on context-usage fetch

The fetch happens in a tokio::spawn with silent error handling (Ok(None) | Err(_) => {}). A transient network error will not affect the main TUI loop and the indicator simply keeps showing the last known value.

Status bar width on narrow terminals

The existing cwd_room calculation already handles truncation when the terminal is too narrow. Because ctx_text is included in the subtraction, on very narrow terminals the cwd will be shortened further before the ctx indicator appears. The ctx indicator itself is never truncated — it appears in full or not at all. On a terminal narrower than about 70 columns, the ctx text could push the cwd to an empty string. This is acceptable given that narrow terminals already have limited status bar space.

Subagent conversations

Subagent conversations are separate Dust conversations with their own IDs. The context-usage fetch is gated on conv_id == app.conversation_id(), so only the main conversation's completion triggers the fetch. Subagent results do not update the indicator.

10 Open Questions

  • Display format: ctx:42% is compact. An alternative is a mini bar [==== ] but that consumes more horizontal space and is harder to implement given the fixed-width span approach. The percentage format is recommended.
  • Separator: Should there be a comma or pipe before ctx: to visually separate it from the subagent count? The existing code uses no separator between skills and subagents — keeping the same convention (a leading space) is consistent.
  • Polling on resume: When a conversation is resumed via the conversation picker, should oxide immediately fetch context usage for that conversation? This is a nice-to-have that can be added in a follow-up. The current plan only fetches after a new agent response.
