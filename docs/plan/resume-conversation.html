Oxide — Implement /resume Slash Command

Reopen and continue an existing Dust conversation from the TUI — Issue #9

Overview

Add a /resume slash command that opens a picker overlay listing the user's past Dust conversations. The user selects one, the full history is fetched and restored into the chat view, and future messages continue in that thread.

User Flow

  • User types /resume and presses Enter
  • A centered popup overlay appears (60% × 60%, exact pattern as agent picker from /switch)
  • Popup title: " Resume Conversation "
  • While fetching, shows: "Fetching conversations..." (yellow)
  • Once loaded, past conversations are listed in reverse chrono order (newest first)
  • Row format per conversation: {title or "(untitled)"} {relative date} — e.g. "Project brainstorm 2 days ago"
  • User can type to filter by title (case-insensitive), and navigate with arrow keys (↑↓)
  • Pressing Enter selects conversation, fetches full history, and closes the popup
  • A system message appears: "Resumed conversation: {title}"
  • New messages sent continue in the restored thread
  • Pressing Esc cancels and returns to Chat mode
  • Footer hints: " ↑↓ navigate Enter select Esc cancel" (same as agent picker)

API Research

Two Dust API endpoints are needed. Both are already supported by the existing auth token.

Method Path Used for
GET /api/w/{wId}/assistant/conversations List past conversations (metadata only — sId, title, created, updated)
GET /api/v1/w/{wId}/assistant/conversations/{cId} Fetch full conversation content (already used internally for agent reply polling)
API endpoint: The list endpoint is under /api/w/ (private API), not /api/v1/w/. The response shape is { conversations: ConversationWithoutContent[] } where each item has sId, title (nullable), created (unix ms), updated (unix ms).
Relative date formatting: When rendering each conversation, compute a human-readable relative date: "2 days ago", "1 hour ago", "just now", etc. from created (unix ms). A simple helper function in ui/picker.rs (or a module util) can do this — check current time with std::time::SystemTime::now().duration_since(UNIX_EPOCH) and diff against created.

Design Decisions

  • Centered popup overlay (60% × 60%): Identical to the agent picker from /switch. Uses Clear widget, centered layout, filter input at top, list in middle, footer hints at bottom. See src/ui/picker.rs::render_picker() as the template.
  • Reuse PickerState pattern: A new AppMode::ResumePicker(ResumePickerState) mirrors AppMode::Picker(PickerState) — same fields (list, filter, selected, loading), same navigation semantics, same `PickerAction` enum dispatch.
  • Two-phase fetching: Phase 1 lists conversations (metadata only, fast). Phase 2 fetches full history only after user selects one (avoids loading all content upfront; same pattern as agent filtering).
  • History reconstruction: The existing ConversationMessage enum only captures agent messages. A UserMessage { content: String } variant is added so both sides of the conversation can be reconstructed from the get_conversation() response.
  • Reverse chronological order: Conversations list is sorted newest first (most recently updated at top) so users find active threads easily.

pub async fn list_conversations(&self) -> Result<Vec<ConversationSummary>> {
    let url = format!("{}/api/w/{}/assistant/conversations", self.base_url, self.workspace_id);
    let response = self.http.get(&url).headers(self.headers()).send().await?;
    let body: ListConversationsResponse = response.json().await?;
    Ok(body.conversations)
}
