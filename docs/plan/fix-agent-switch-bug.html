Oxide Fix Agent Switch Bug

Goal: Ensure that when a user runs /switch and selects a new agent, the next message is sent to that new agent — not the original one.
Scope: Two files: src/main.rs (function signature fix) and src/app.rs (filter reset fix + new test). No API changes, no UI changes.
Success: Sending a message after /switch reaches the newly selected agent. Unit test asserts app.agent_id persists correctly after the picker selection path.

1 Overview

The /switch slash command opens the agent picker modal. Users type to filter agents, press Enter to select, and expect the next message to go to the chosen agent. Instead, it silently sends to the original agent.

The bug has two independent components:

  • Primary bug — handle_agent_picker_key_event() receives a cloned Option<DustClient>. The clone's set_agent() mutates the clone, not the client variable that lives in the main loop. The main loop's client retains the old agent_id forever.
  • Secondary bug — set_picker_filter() already resets state.selected = 0 (lines 340–342), which is correct. However, there is no reset when agents load while the user has already moved the selection. set_picker_agents() at line 334 does reset selected = 0, so this path is safe. The issue is when the user types quickly before agents arrive: they could move selection on an empty list, then the filter reset leaves selected stale. Adding an explicit selected = 0 on every filter change is the right defensive fix.

What is NOT broken

  • app.switch_agent() correctly updates app.agent_id and app.agent_name (line 400–405 of src/app.rs).
  • The "Switched to agent-name" system message is pushed correctly.
  • The Dust API itself does not need to be changed.

2 Root Cause Analysis

Bug 1 — Cloned client loses mutation (Primary)

In src/main.rs, the main loop calls (line 874):

handle_agent_picker_key_event(key, &mut app, client.clone());

client.clone() produces an Option<DustClient> that is passed by value into the function signature: 

fn handle_agent_picker_key_event(
    key: crossterm::event::KeyEvent,
    app: &mut App,
    client: Option<DustClient>,   // <-- owned clone, not &mut
) {

On the PickerAction::Select branch (lines 530–540), the function does: 

if let Some(mut c) = client {
    c.set_agent(&agent_id);
}

This moves the clone out of the Option, mutates it, and then it is dropped at end-of-scope. The original client: Option<DustClient> variable in run_tui_main_loop is untouched. Every subsequent call to handle_pending_message_submit uses client.clone() of the unchanged original — still carrying the old agent_id.

Bug 2 — Off-by-one after filter changes (Secondary)

set_picker_filter() already contains state.selected = 0, so this is already fixed in the current code. The secondary bug mentioned in the issue description is therefore a pre-existing correct implementation — the real risk is that picker_move_selection() computes count from the filtered list, and if selected is out of bounds for the current filter, filtered.get(selected) on the Select branch returns None, silently no-oping the selection. The defensive fix (resetting selected = 0 in set_picker_filter) is already present. We still add a unit test to pin this behaviour.

Fix path

run_tui_main_loop owns client: Option<DustClient> | v handle_agent_picker_key_event(key, &mut app, &mut client) | v PickerAction::Select app.switch_agent("new-id", "new-name") // app.agent_id updated if let Some(c) = client { c.set_agent("new-id") } // mutates original | v next frame handle_pending_message_submit(content, client.clone(), ...) | client.agent_id is now "new-agent" ✓ v client.send_message_flow_with_skills(...) // sent to new agent!

Step 5 — Add integration tests in src/app.rs

test src/app.rs

Add two tests inside the existing #[cfg(test)] mod tests block (after line 1349):

Test A — switch_agent updates agent_id field

#[test]
fn switch_agent_updates_agent_id() {
    let mut app = App::new("old-agent", "/workspace", None);
    app.switch_agent("new-id", "new-name");
    // agent_id must reflect the new selection so future HTTP requests use it
    assert_eq!(app.agent_id, "new-id");
    assert_eq!(app.agent_name(), "new-name");
}

Note: app.agent_id is a private field. Add a pub fn agent_id(&self) -> &str accessor on App (guarded by #[cfg(test)] mirrors the existing DustClient::agent_id pattern, or make it unconditionally public — see Risk section). Alternatively test through observable side-effects already present.

Test B — set_picker_filter resets selected index

#[test]
fn set_picker_filter_resets_selected_index() {
    let mut app = App::new("a", "/workspace", None);
    app.enter_picker();
    app.set_picker_agents(vec![
        AgentInfo { s_id: "a1".into(), name: "alpha".into(), description: "".into(), scope: "".into() },
        AgentInfo { s_id: "a2".into(), name: "beta".into(),  description: "".into(), scope: "".into() },
    ]);
    app.picker_move_selection(1);
    assert_eq!(app.picker_selected(), 1);

    // Typing a filter character must reset selection to 0
    app.set_picker_filter("alpha");
    assert_eq!(app.picker_selected(), 0,
        "set_picker_filter must reset selected index to prevent off-by-one selection");
}

Test C — switch_agent_after_filter selects correct agent

#[test]
fn switch_agent_persists_after_filter() {
    let mut app = App::new("original", "/workspace", None);
    app.enter_picker();
    app.set_picker_agents(vec![
        AgentInfo { s_id: "a1".into(), name: "alpha".into(), description: "".into(), scope: "".into() },
        AgentInfo { s_id: "a2".into(), name: "beta".into(),  description: "".into(), scope: "".into() },
    ]);
    app.set_picker_filter("beta");
    let filtered = app.picker_filtered_agents();
    // selected == 0 after filter reset, first (and only) result is beta
    let agent = filtered.get(app.picker_selected()).unwrap();
    app.switch_agent(&agent.s_id, &agent.name);

    assert_eq!(app.agent_name(), "beta");
    // Ensure app-level agent id is updated (visible through push_system_message content)
    let last = app.messages().last().unwrap();
    assert!(last.content.contains("beta"));
}

6 Risk Assessment

Risk 1 — Borrow checker conflict in the select! block

The main loop's tokio::select! block (line 862–926) uses client in multiple arms:

  • Line 865: if let Some(ref mut c) = client — already &mut
  • Line 874: proposed &mut client — new mutable borrow
  • Line 884: client.clone() — cloned, no borrow
  • Line 889: client.clone() — cloned, no borrow
  • Line 896: client.as_ref() — shared borrow
  • Line 916: client.as_ref() — shared borrow
  • Line 930: client.clone() — cloned, no borrow

Because tokio::select! only executes one arm per iteration, borrows do not overlap at runtime. However, the Rust borrow checker analyses them statically within the macro expansion. The existing client.clone() pattern was used precisely to avoid this — switching to &mut client for only the Picker arm should compile cleanly because only one arm is active per iteration and the macro generates separate scopes. If a borrow conflict does occur at compile time, the fix is straightforward: extract the mutable borrow into a helper that takes &mut Option<DustClient> and is called outside the select block (moving the dispatch earlier).

Mitigation: Run cargo check immediately after the signature change. If it fails, the implementer should move the picker key handling outside the select! block by staging the action and executing it after the select!.

Risk 2 — agent_id field is private in App

Test A requires reading app.agent_id. This field has no public getter. Two options:

  • Option A (recommended): Add pub fn agent_id(&self) -> &str { &self.agent_id } to App. This is a clean public API that future features will also want.
  • Option B: Test through agent_name() only, since switch_agent also updates agent_name. Simpler but tests less of the contract.

The existing DustClient::agent_id() is already #[cfg(test)]-gated — follow the same pattern if adding to App.

Risk 3 — No Dust API surface is affected

The fix is entirely within the TUI layer. DustClient::set_agent() already exists and works correctly. No HTTP calls, no serialisation changes, no SSE stream changes.

Risk 4 — handle_resume_picker_key_event is unaffected

The parallel handle_resume_picker_key_event function also receives client: Option<DustClient> by value, but it spawns a tokio::spawn that moves client into the async block — not calling set_agent(). No bug exists there. Do not change its signature.
