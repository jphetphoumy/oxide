Oxide Local Skills Support

Issue #12 · feat(skills)

Goal: Discover, load, and activate skill definitions from local disk; inject skill context into Dust messages as a session-level system prompt.
Scope: Local file discovery, YAML skill format, /skill slash command, modal picker, session-scoped activation, status bar indicator, unit tests.
Out of scope: Remote skill registry sync, agentskills.io network fetching, per-message (ephemeral) skill activation, skill authoring wizard.
Success: User can run /skill, pick a skill from disk, and have its instructions injected as a preamble into the first message of a new conversation.

1 Skill File Format

Skills are .md files with YAML frontmatter, stored in ~/.local/share/oxide/skills/. This matches the agentskills.io convention and keeps skills human-readable in any editor.

---
name: Code Reviewer
description: Senior engineer focused on correctness and simplicity
version: "1.0"
---
You are a senior software engineer reviewing a pull request.

Focus on:
- Correctness bugs and logic errors
- Simplification opportunities
- Security issues

Be concise. Format findings as a bullet list with severity tags (critical / minor).

Required frontmatter fields

  • name — display name in picker
  • description — one-line subtitle

Optional frontmatter fields

  • version — informational only
  • author — informational only
  • tags — future filtering (ignored for now)

Parsing strategy

Split on the second --- delimiter. Parse the header block as YAML using serde_yaml. Everything after is the raw instructions body (trimmed). Files without valid frontmatter are silently skipped during discovery with a tracing::warn!.

2 Discovery & Loading

Discovery path

Skills are loaded from .agents/skills/ relative to the current working directory, following the agentskills.io spec. This path is canonical and not configurable.

├── code-review.md
├── research-assistant.md
└── sql-expert.md

Error handling

  • Missing directory → return empty Vec (not an error; user may not have any skills yet)
  • Unreadable file → tracing::warn! and skip
  • Missing name or description → skip with warning
  • No serde_yaml parse errors propagated to the UI — load as many as possible

3 Config Changes

No config changes needed

The discovery path .agents/skills/ is canonical per the agentskills.io spec. No [skills] section is added to config.toml. config.rs is unchanged.

4 UX: Slash Command & Picker

Slash command format: /skills:<id>

Each skill gets its own slash command derived from its filename stem. A file code-review.md registers as /skills:code-review. The skills: namespace avoids collision with built-in commands and makes skills discoverable as a group.

FileCommandTab completes from
code-review.md/skills:code-review/sk, /skills, /skills:co
sql-expert.md/skills:sql-expert/skills:sq
research.md/skills:research/skills:re

Dynamic registration: slash.rs must go runtime

The current COMMANDS is a &[SlashCommandDef] static array. Skills are loaded from disk at runtime, so the command list must become dynamic. Change COMMANDS to a runtime-owned Vec<SlashCommandDef> (or add a separate skill_commands list) loaded once at startup alongside skill discovery.

Startup skill discovery

Skills are discovered once at startup (not on each /skills:* invocation). The discovered list is stored on App and used to both populate dynamic slash commands and map command IDs to Skill structs.

// main.rs — at startup, before event loop
let skills = skills::discover_skills(&config.skills_path());
app.set_skills(skills.clone());
slash::register_skill_commands(&skills); // adds /skills:<id> entries

No picker needed

Because each skill is its own command, no modal picker is required. Tab completion in the existing inline command menu surfaces all /skills:* entries. The AppMode::SkillPicker mode and render_skill_picker() are dropped from scope.

Active skills on App

// app.rs
pub struct App {
    // ... existing fields ...
    pub skills: Vec<Skill>,          // loaded at startup from .agents/skills/
    pub active_skills: Vec<Skill>,   // set by /skills:<id>, cleared on /new
}

Flow diagram

Startup → discover_skills() → app.set_skills() + register_skill_commands() │ ▼ User types /sk → inline menu shows all /skills:* entries (tab completes) │ ▼ User submits /skills:code-review │ ▼ parse_slash_command() → SlashCommand::ActivateSkill("code-review") │ ▼ app.activate_skill("code-review") → active_skill = Some(skill) │ ▼ Status bar: [skill: Code Reviewer] — next /new will inject it

5 Built-in MCP Tool: oxide_skill

Why MCP, not message injection

  • contentFragments tested and rejected — Dust surfaces its own native skills, not injected content.
  • Full-body prepend — duplicates skill content into history on every conversation. Wasteful.
  • File system MCP — requires the bash tool to be separately configured.
  • Built-in MCP tool — the agent calls oxide_skill(skill_id) and gets the content on demand. No other tool needed. Same pattern as oxide_bash.

Tool registration — same pattern as oxide_bash

Auto-registered in McpManager::init() with no config entry required. Always available.

// src/mcp/mod.rs — McpManager::init()
tools.push(McpTool {
    name: "oxide_skill".to_string(),
    description: "Load the full instructions for a local skill from .agents/skills/".to_string(),
    input_schema: serde_json::json!({
        "type": "object",
        "properties": {
            "skill_id": {
                "type": "string",
                "description": "Skill id, e.g. \"code-review\" → .agents/skills/code-review.md"
            }
        },
        "required": ["skill_id"]
    }),
});

// src/mcp/mod.rs — McpManager::call_tool()
if tool_name == "oxide_skill" {
    let skill_id = input.get("skill_id").and_then(|v| v.as_str())
        .context("oxide_skill requires 'skill_id'")?;
    let path = PathBuf::from(".agents/skills").join(format!("{skill_id}.md"));
    let content = std::fs::read_to_string(&path)
        .with_context(|| format!("skill '{skill_id}' not found at {}", path.display()))?;
    return Ok(ToolResult { content, is_error: false, tool_use_id: String::new() });
}

First message: lightweight index only

When skills are active, a compact index is prepended to the first user message. Name and description only — no file content. The agent calls oxide_skill to load whatever it needs.

# Oxide local skills
code-review: Help review code for correctness, simplicity, and security
sql-expert: Optimize SQL queries and schema design

Use oxide_skill(skill_id) to load a skill's full instructions.

User activates /skills:code-review → types first message → Enter │ ▼ create_conversation_body() — active_skills not empty? ├── yes → prepend skills index (name + description, all active skills) └── no → first message unchanged │ ▼ POST /conversations │ ▼ Agent calls oxide_skill("code-review") │ ▼ McpManager::call_tool("oxide_skill") → reads .agents/skills/code-review.md → returns content │ ▼ Agent applies skill instructions

Status bar indicator

When skills are active, the status bar shows all active skill names:

Rendered in var(--purple). /new clears all active skills.

8 Testing Strategy

Unit tests (Phase 1–4)

  • discover_skills with empty dir → empty vec
  • discover_skills with valid .md → correct Skill
  • File missing name field → skipped
  • File with no frontmatter → skipped
  • Config skills_path() defaults to XDG dir
  • Config custom path with ~ is expanded
  • /skill tab-completes
  • /skill submit → SlashCommand::Skill
  • Mode transition: enter_skill_picker sets loading, select_skill sets active
  • /new clears active_skill
  • create_conversation_body with active skill: first message has <skill> preamble
  • create_conversation_body without skill: first message is unchanged
  • post_message_body never injects skill (even if active)

Manual TUI tests (tmux)

  • Type /sk → tab-completes to /skill
  • Submit /skill → modal opens
  • Up/Down navigates skill list
  • Type to filter list
  • Enter selects → status bar shows skill name
  • Esc dismisses → no skill selected
  • Send message with active skill → instructions prepended (visible in Dust)
  • /new clears skill indicator
  • Empty skills dir → modal shows "No skills found"

9 Risks & Mitigations

Skill instructions may be long

Very long skill bodies will consume Dust context budget on every message. Mitigation: no automatic truncation in v1 — document that skill files should be concise. Add a tracing::warn! if instructions exceed 4 KB.

serde_yaml dependency

Adds a crate for YAML parsing. Mitigation: evaluate whether a manual frontmatter splitter (split on ---, parse key-value pairs) avoids the dependency. Only add serde_yaml if the manual approach proves fragile.

agentskills.io format compatibility

The agentskills.io schema is not formally documented. The frontmatter fields chosen (name, description, version) are a common subset. Unknown fields are ignored by serde, so future additions won't break loading.

10 Decisions

Activation scope: per conversation

Skills are activated per conversation, not per session. active_skills is cleared on /new. /resume does not restore skills from the previous conversation.

Multiple active skills: yes

Multiple skills can be active simultaneously. App holds active_skills: Vec<Skill>. Each /skills:<id> invocation appends to the list (or is a no-op if already active). The injection block lists all active skills.

Injection format: compact index block

The preamble is a compact heading + table — one line per skill, name and description only. The agent knows to read the file path for full instructions.

# Oxide local skills
code-review: Help review code for correctness, simplicity, and security
sql-expert: Optimize SQL queries and schema design

{user's first message}

The path to each skill file (.agents/skills/<id>.md) is derivable by the agent from the id. No explicit path needed in the index if the agent knows the convention. Alternatively include it for clarity:

# Oxide local skills
code-review (.agents/skills/code-review.md): Help review code for correctness, simplicity, and security
