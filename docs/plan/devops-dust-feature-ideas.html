Oxide DevOps-Oriented Dust Feature Ideas

Goal: Identify Dust API capabilities that Oxide does not currently expose and rank them by value for DevOps/SRE/platform engineers working from the terminal.
Scope: Feature discovery, API gap analysis, ranked prioritization, and implementation guidance for native file attachments.
Success: Clear roadmap with validated first feature (native file attachments) and phased follow-on work.

1 Current Oxide Baseline

Oxide already supports interactive chat with agent switching, conversation resume, local skills injection, and MCP/subagent execution with tool approval. The current implementation covers:

  • Single-agent chat loop in src/main.rs
  • Built-in slash commands: /new, /switch, /resume
  • Dust API client for conversations, SSE streaming, agent listing, and MCP result submission
  • Message rendering with role-based colors and approval UI

2 Dust Capability Gaps

The Dust codebase exposes several capabilities that Oxide does not currently use:

  • File attachments — Upload and embed artifacts (logs, manifests, YAML)
  • Mention suggestions — Multi-agent discovery and handoff
  • Toolset mutation — Add/remove tools per conversation
  • Blocked action recovery — List and retry failed tool calls
  • Structured questions — Multi-choice and typed agent responses
  • Message feedback & retry — Rate answers and re-run messages
  • File download — Pull generated artifacts locally

3 Ranked Feature Ideas

Native File Attachments

Attach logs, manifests, runbooks, and incident artifacts directly instead of pasting them into the prompt.

Complexity: Medium Very High Value

Action Timeline & Recovery

See which tools failed, why they blocked, and retry only blocked actions instead of restarting everything.

Complexity: Medium High Value
3

Structured Question Answering

Answer agent prompts with multi-choice or typed responses natively instead of switching to web.

Complexity: Medium–High High Value
4

Agent Mentions & Handoff

Pull other agents into the conversation thread during incidents without losing context.

Complexity: Medium Medium–High Value
5

Conversation-Scoped Tools

Add or remove AWS/K8s/GitHub tools per incident without restarting.

Complexity: High Medium–High Value
6

Artifact Export & Download

Download generated configs, reports, or transformed files locally without copy/paste.

Complexity: Medium Medium Value
7

Feedback & Retry Ergonomics

Lightweight rating and retry controls for iterating on prompts and validating new agents.

Complexity: Low–Medium Medium Value

4 Recommended First Feature: Native File Attachments

Why This Feature

  • Clearest DevOps payoff — artifact-heavy workflows benefit immediately
  • Well-defined API — mirrored in Dust SDK with proven patterns
  • Fits current model — extends existing chat without major rewrites
  • Unlocks follow-on work — downloaded artifacts, richer rendering, incident bundles

Target User Experience

# Inline attachment syntax (primary)
Summarize the failures: @/var/log/kubectl.log

# Or list staged files
/attachments

# Remove a file
/detach /var/log/kubectl.log

# Alternative slash command for explicit staging
/attach /var/log/kubectl.log

# Then send without re-typing the path
Here's the context needed

Behavior: Inline @ syntax stages files for the current message. Staged files persist until send succeeds or user calls /new, at which point they clear.

5 Implementation Scope

Scope: File staging, slash commands, Dust client methods, and attachment-aware message sending for both new and resumed conversations.
Out of scope (v1): Glob patterns, immediate uploads, visually attached historical messages, directory support.

File-by-File Changes

File Changes
src/app.rs Add staged attachment model with path, name, size; add methods: stage_attachment(), remove_attachment(), clear_attachments()
src/handler.rs Extend SlashCommand enum with Attach(String), Detach(String), Attachments; update parser for arguments
src/slash.rs Register /attach, /attachments, /detach with terminal-friendly descriptions
src/dust/types.rs Add serde types: file upload request/response, content fragment request/response
src/dust/client.rs Add methods: upload_file(), post_content_fragment(), send_message_flow_with_attachments()
src/main.rs Handle slash commands, call new client methods, clear attachments after successful send
src/ui/messages.rs Render staged attachments as compact system-style block near input when pending

Dust API Flow

User calls /attach <path> | v POST /api/v1/w/{wId}/files (multipart upload) | v POST /api/v1/w/{wId}/assistant/conversations/{cId}/content_fragments | v POST /api/v1/w/{wId}/assistant/conversations/{cId}/messages (with content fragments) | v Agent receives file context

6 Testing Strategy

Unit Tests

  • Slash command parsing for /attach, /detach, /attachments
  • App staged attachment state transitions (add, remove, clear)
  • Request serialization for file upload and content fragments
  • URL builders for new Dust client methods

Integration-Style Tests

  • Multipart upload request setup
  • Branching logic between new conversation and existing conversation flows

Manual tmux Test

  1. Start cargo run in detached tmux session
  2. Attach a small text file with /attach
  3. Verify staged attachment renders in UI
  4. Send a prompt asking agent to summarize the file
  5. Verify upload succeeds and staged list clears after send

7 Implementation Milestones

  1. Attachment slash commands & app state — Register commands, add staging model
  2. Dust client methods — File upload, content fragment, serde types
  3. Attachment-aware send flow — Wire into main.rs, handle both new and resumed conversations
  4. UI rendering — Display staged attachments in chat
  5. Testing & validation — Unit tests, manual tmux pass

8 Follow-On Sequence

After native file attachments are complete, recommended priority order:

  1. Action timeline + blocked-action retry — Visibility into tool failures, recovery paths
  2. Structured question answering — Multi-choice and typed agent responses
  3. In-conversation agent mentions — Multi-agent threads without context loss
  4. Conversation-scoped toolset toggling — Dynamic tool add/remove per incident
  5. Artifact download — Pull generated files locally

9 Open Decisions

  • Glob support in v1? — Recommended: exact paths only for now
  • Upload timing: Immediate on /attach or deferred until send? — Recommended: defer to avoid orphan uploads
  • Persistent visual attachment in v1? — Recommended: not required; defer to later phase
  • Directories or just files? — Recommended: regular files only in v1
