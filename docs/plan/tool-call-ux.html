Oxide Tool Call UX

Goal: Show tool calls inline in the chat message history, with tool name + arguments and the result after execution — matching the Claude Code terminal UX.
Scope: New Role::ToolCall variant, inline rendering in messages.rs, wiring in app.rs and main.rs. The existing approval overlay is simplified to just keybinding hints.
Success: Every tool call (auto-approved or manual) leaves a persistent ● Tool(arg) line in history, followed by ⎿ result after execution. Denied calls show ✗ denied.

2 Data Model changes — app.rs

New types

pub enum ToolCallStatus {
    Pending,   // waiting for user approval
    Running,   // approved, executing
    Done,      // finished successfully
    Failed,    // execution error
    Denied,    // user denied
}

pub struct ToolCallEntry {
    pub call_id: String,              // tool_call.id
    pub tool_name: String,
    pub input: serde_json::Value,
    pub status: ToolCallStatus,
    pub result: Option<String>,       // stdout / error shown under ⎿
    pub expanded: bool,               // ctrl+o toggles full result view
    pub started_at: std::time::Instant,
    pub finished_at: Option<std::time::Instant>,
}

// Truncation threshold: show first N output lines by default
const TOOL_RESULT_PREVIEW_LINES: usize = 5;

Header line format

  ● ToolName(first_arg_value)
         │         │
         │         └── extracted from input JSON:
         │              • if object has "command" key → show its string value
         │              • if object has exactly one key → show that value
         │              • otherwise → show truncated JSON (max 40 chars)
         └── bullet colored by status

Max header line: truncate arg preview at 60 chars with ….

Result rendering — truncation

Result text is split on \n. Each line is word-wrapped with the existing wrap_line() helper.

  • First result line gets the ⎿ prefix; subsequent lines get a indent (3 spaces) to align under the arrow.
  • If collapsed (expanded == false) and the result has more than TOOL_RESULT_PREVIEW_LINES (5) lines: render only the first 5, then append a dim hint line: … +N lines (ctrl+o to expand)
  • If expanded (expanded == true): render all lines, then append a dim hint: (ctrl+o to collapse)
  • Results with ≤ 5 lines: rendered in full, no hint shown.

Simplified approval overlay

The tool_approval_lines() function in render_messages currently renders the full tool detail + keybindings as a bottom overlay. With inline display, the overlay is removed entirely — the Pending state of the inline ToolCall entry already shows [y] approve [n] deny [Esc] cancel.

The AppMode::ToolApproval block in render_messages is deleted.

ctrl+o targeting

ctrl+o expands/collapses the most recent ToolCall message in history (regardless of scroll position). This is the same heuristic Claude Code uses — the last tool output is the one you care about. Future work could add per-entry selection, but that's out of scope here.

Implementation: app.last_tool_call_id() walks messages in reverse, returns the first Role::ToolCall entry's id. The key handler calls app.toggle_tool_call_expanded(&id).

4 Wiring changes — main.rs

Auto-approve path (handle_tool_use_event / handle_mcp_tool_use_event)

  1. Call app.push_tool_call(tool_call.clone()) → get call_id
  2. Call app.set_tool_call_running(&call_id) immediately (no user gate)
  3. Spawn async task; on success call app.complete_tool_call(&call_id, result) via a DustEvent::ToolCallResult message
  4. On error call app.fail_tool_call(&call_id, error) the same way

Manual approve path (enter_tool_approval callers)

  1. Call app.push_tool_call(tool_call.clone()) → call_id
  2. Call app.enter_tool_approval(tool_call, call_id) — now carries the id
  3. On approve: set_tool_call_running, spawn, send result via event
  4. On deny: app.deny_tool_call(&call_id)

7 Risks & mitigations

Async result delivery

The tool executes in a spawned task; the result arrives asynchronously via DustEvent::ToolCallResult. If the conversation changes (new conversation, agent switch) before the result arrives, the call_id lookup in messages will simply not find a match and silently no-op — no crash, stale result just disappears with the cleared messages.

Long result output

Tool results are stored in full in ToolCallEntry.result, but only the first 5 lines are rendered by default. The … +N lines (ctrl+o to expand) hint tells the user there's more. ctrl+o re-renders the full output inline. This keeps the message history scannable without losing data.

Backward compatibility

Role::ToolCall is a new enum variant — all existing match arms have a wildcard or exhaustive match, so the compiler will flag any missing cases at compile time. No silent regressions possible.
