Oxide Dust API Integration Plan

Goal: Replace the echo agent with real Dust API calls so Oxide can talk to any Dust agent — create conversations, send messages, and stream responses token-by-token.
Prerequisite: The TUI chat shell from feat/tui-chat is merged.
Reference: Dust CLI (dust-cli) and @dust-tt/client SDK, plus the Dust OpenAPI spec (front/public/swagger.json).

0 Context & Approach

The official Dust CLI is a TypeScript/Ink app using the @dust-tt/client SDK. Oxide is a Rust TUI — we build our own HTTP client against the same REST API.

What we take from the Dust CLI

  • Origin context: messages include origin: "cli" in the context object, so Dust analytics can distinguish CLI traffic from web traffic.
  • User-Agent header: "User-Agent: Dust CLI" — same as the official CLI, used by Dust for billing and usage tracking.
  • API structure: same REST endpoints, same JSON shapes, same SSE streaming protocol.
  • Default agent config: read ~/.config/oxide/config.toml and use [dust].agent_id when no OXIDE_AGENT_ID override is set.
  • Conversation title: new conversations use the CLI prefix CLI Question: <first 30 chars>..., matching the official client.
  • Workspace selection: after login, fetch /api/v1/me, present the user's workspaces, and store the selected workspace_id for later API calls.

What we do differently

  • No MCP server registration (out of scope for this phase).
  • Rust-native SSE parsing instead of eventsource-parser.
  • Rust-native keyring via keyring crate instead of Node's keytar.
  • JWT decoding via jsonwebtoken crate instead of jwt-decode.

1 API Endpoints

All endpoints are relative to the base URL (e.g. https://dust.tt) and scoped to /api/v1/w/{workspaceId}/assistant/.

List agents

GET /api/v1/w/{wId}/assistant/agent_configurations?view=list

Returns available agent configurations. We use this to let the user pick an agent or validate a configured agent ID.

Create conversation

POST /api/v1/w/{wId}/assistant/conversations 
{
  "title": "CLI Question: <first 30 chars>...",
  "visibility": "unlisted",
  "message": {
    "content": "user message text",
    "mentions": [{ "configurationId": "agent_sId" }],
    "context": {
      "timezone": "Europe/Paris",
      "username": "jphetphoumy",
      "fullName": "...",
      "email": "...",
      "origin": "cli"
    }
  }
}

Creates a new conversation with the first user message. Returns conversation.sId and message.sId (the user message ID, needed to find the agent reply for streaming). The CLI sets the title itself before sending the request.

Post follow-up message

POST /api/v1/w/{wId}/assistant/conversations/{cId}/messages

Same body shape as the message object above. Used for subsequent messages in an existing conversation.

Stream agent answer events (SSE)

GET /api/sse/v1/w/{wId}/assistant/conversations/{cId}/messages/{mId}/events

Server-Sent Events stream. Key event types:

Event typeDescription
generation_tokensText tokens (classification: tokens or chain_of_thought)
agent_message_successAgent finished — stream complete
agent_errorAgent execution error
user_message_errorMessage creation failed
agent_generation_cancelledGeneration was cancelled

Get conversation

GET /api/v1/w/{wId}/assistant/conversations/{cId}

Retrieve full conversation (needed after posting a follow-up message to find the agent message ID for streaming). Oxide polls this endpoint until the agent message appears, then streams /messages/{agentMessageId}/events.

2 Authentication & Headers

Two authentication modes (same as Dust CLI)

1. OAuth Device Flow (interactive — primary)

Same WorkOS device flow as the official Dust CLI. The user runs oxide login once.

  1. Initiate: POST https://api.workos.com/user_management/authorize/device with client_id and scope=openid profile email. Returns device_code, user_code, verification_uri_complete, interval.
  2. Display: Show the user_code in the terminal, open verification_uri_complete in the browser.
  3. Poll: Every interval seconds, POST to https://api.workos.com/user_management/authenticate with grant_type=urn:ietf:params:oauth:grant-type:device_code and the device_code. Handle authorization_pending (keep polling) and slow_down (back off +5s).
  4. Token received: Store access_token and refresh_token in the OS keyring. Decode the JWT to extract the region from the https://dust.tt/region claim.
  5. Workspace selection: Call GET /api/v1/me, read the user's workspaces list, prompt if there is more than one, and store the chosen workspace_id in the keyring.

Token refresh

Before each API call, check JWT expiry. If within 30 seconds of expiring, refresh via:

POST https://api.workos.com/user_management/authenticate
Content-Type: application/x-www-form-urlencoded

grant_type=refresh_token&client_id={CLIENT_ID}&refresh_token={stored_refresh_token}

On 400/401, clear tokens and prompt re-login.

Message context (origin)

Every message body includes a context object with "origin": "cli". This is what the Dust CLI sends and is how Dust tracks CLI usage. We keep the same value so our traffic is recognized as CLI-originating.

Streaming payload shape

The SSE transport uses Dust's envelope shape, where each event frame contains an eventId and a nested data payload. Oxide unwraps that envelope before deserializing the inner Dust event object.

// Per-request: token may have been refreshed

7 Step 3 — SSE Stream Parser

src/dust/stream.rs

Parse the SSE byte stream from reqwest into typed events. The SSE protocol is simple: lines prefixed with data: , events separated by \n\n.

pub struct EventStream {
    response: reqwest::Response,
    buffer: String,
}

impl EventStream {
    /// Yields the next parsed event from the SSE stream.
    /// Returns None when the stream ends.
    pub async fn next_event(&mut self) -> Option<Result<StreamEvent>> {
        loop {
            // Try to extract a complete event from buffer
            if let Some(pos) = self.buffer.find("\n\n") {
                let event_text = self.buffer[..pos].to_string();
                self.buffer = self.buffer[pos + 2..].to_string();

                // Extract data lines
                let data: String = event_text.lines()
                    .filter_map(|l| l.strip_prefix("data: "))
                    .collect::<Vec<_>>()
                    .join("\n");

                if data.is_empty() { continue; }

                match serde_json::from_str::<StreamEvent>(&data) {
                    Ok(event) => return Some(Ok(event)),
                    Err(e) => return Some(Err(e.into())),
                }
            }

            // Read more bytes from the response
            match self.response.chunk().await {
                Ok(Some(chunk)) => {
                    self.buffer.push_str(
                        &String::from_utf8_lossy(&chunk)
                    );
                }
                Ok(None) => return None,  // stream ended
                Err(e) => return Some(Err(e.into())),
            }
        }
    }
}

Message flow

  1. User presses Enter → Action::Submit
  2. Push user message to messages immediately (optimistic UI)
  3. Push an empty agent message placeholder, set is_streaming = true
  4. Spawn a tokio task:
    • If conversation_id is None → create_conversation()
    • Else → post_message() + get_conversation() to find agent message ID
    • Then stream_events(), sending tokens over an mpsc channel
  5. Main loop receives tokens via channel → appends to last agent message → re-renders
  6. On agent_message_success or error → set is_streaming = false

Channel design

enum DustEvent {
    Token(String),             // append text to current agent message
    StreamDone,                // agent finished
    Error(String),             // show error in UI
    ConversationCreated(String), // store conversation_id
}

The main event loop selects on both crossterm events and the DustEvent channel, keeping the UI responsive while streaming.
